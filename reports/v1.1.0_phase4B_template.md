## Why Haiku workers are not spawned

The brief directs: "DISMANTLE_TCB_TRACE=1 — USE IT to direct optimization choices."
The trace shows MoE at 0.1% of time. Implementing `v2t_batch` variants for all
three quant types via Haiku workers would:
- Add ~300 lines of Metal + Rust code per variant
- Produce zero measurable e2e improvement (MoE is 0.28ms/token vs 60ms total)
- Create maintenance surface for dead-code paths

Per the brief's own directive, TCB trace data overrides the pre-planned 4B work.

## Recommendation for Phase 5

The real bottleneck is `tcb_commit` (69.4%) and `gemv_f16` LM head (30.3%).
Phase 5 should target:
1. LM head: top-K vocab filter before `gemv_f16` (only compute logits for top-K tokens)
2. TCB commit amortization: multi-layer fused TCB to reduce commit count
3. After Phase 4C+4D (batched verify) land, re-measure — the bottleneck
   distribution will shift when K=4 tokens go through one forward pass.
