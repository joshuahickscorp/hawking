## What was shipped

### Phase A — Mixtral TCB port (commit d38efcb)
Reduced Metal command buffer commits from 97/token to 65/token (3→2 commits/layer).
Restructured `forward_token_tcb` into TCB1 (attn+gate) / CPU topk / TCB2 (experts+accum) pattern.
Added GPU KV cache (512-token cap, 128MB) to avoid per-token mmap reads on K/V projections.
Added `mha_decode_f32` Metal kernel (GQA decode) and `moe_accum_2_scaled` Metal kernel.

**Outcome:** No measurable dec_tps improvement. Root cause diagnosed: expert weight pages (~24GB of 26GB GGUF) cannot fit in 18GB RAM. OS evicts pages between tokens, forcing SSD reads at ~1GB/s. TCB2 (expert FFN) takes ~188ms/layer × 32 = ~6000ms per token; GPU compute budget at 200GB/s would be ~1ms.

### Phase C — N-gram speculative decoding (commit 2b5dbd2)
`NGramDraft` struct: backward linear scan over token history, proposes up to k tokens.
New `SpeculateMode::NGram` variant, wired into V2-Lite generate loop.
`--speculate ngram` CLI flag. 5 unit tests passing.

**Outcome:** 100% acceptance rate on repetitive prompts, identical dec_tps to baseline.
Root cause of 1× speedup: serial verification requires one full forward pass per accepted token (KV cache update is mandatory). Batched verification would yield 1.2-1.4×; not implemented.

### Phase F — Mixtral sync page pre-warming (commit ae45309)
Added synchronous page-touch loop in `MixtralEngine::load` after all layers are constructed.
Reads one byte per 4KB page of `attn_q` + `attn_output` across all 32 layers (~1.85GB).
Complements the async `madvise_willneed_mmap` already in place.

**Outcome:** No measurable improvement. Expert weight pages (not attn projections) are the bottleneck. Prefill reads all active experts, evicting any pages pre-warmed for attn. Phase F is architecturally correct but not load-bearing until expert weights fit in RAM.

### Phase B — Q4_K_M llama.cpp kernel port
**Not attempted.** Expert FFN SSD I/O dominates at ~188ms/layer; a faster GEMV kernel saves <1ms/call. Compute optimization is irrelevant while the bottleneck is memory bandwidth from SSD.

## Root cause of Mixtral plateau

The 26GB Mixtral GGUF cannot fit in 18GB RAM. Per-token decode activates 2 of 8 experts per layer × 32 layers = 64 expert FFN calls. Each expert is ~200MB of Q4_K_M weights. At 1GB/s effective SSD bandwidth, page-fault cost per token ≈ 64 × 200MB × (miss rate) → dominates decode.

**What would unblock ≥1.0 dec_tps on 18GB:**
1. Expert pinning: keep the top-2 most-accessed experts permanently in a pinned Metal buffer. Requires `expert_access_counts` profiling run + static routing assumption. Reduces hot-path faults from 64 to 0 for cached experts.
2. Expert prefetch: after gate inference (CPU-side topk), `madvise_willneed` the next token's experts before committing TCB2. Requires async prefetch pipeline.
3. Smaller model: Q2_K quantization (~13GB) might fit in RAM with the KV cache compressed; not tested.
