**Date:** 2026-05-05
**Status:** STUDY-ONLY — no shader changes in this wedge

## 1. Threadgroup geometry

### llama.cpp `kernel_mul_mv_q4_K_f32_impl` (line 7716)

```
TG threads:  NSG × 32   (NSG = FC_mul_mv_nsg, Metal function constant; default 4)
TG output:   NSG × nr0 rows  (nr0 = N_R0_Q4_K = 2 for Q4_K)
             = 4 × 2 = 8 rows per TG, 128 threads per TG
Grid:        ceil(rows / 8) × ceil(ne01/4) × batch
```

Within each simdgroup (32 threads):
- `ix = tiisg / 8` → 0..3: which of 4 8-thread sub-groups
- `it = tiisg % 8` → 0..7: lane within sub-group
- Each sub-group covers distinct 64-element chunks of the Q4_K block (256 elements / 4 sub-groups)
- All 4 sub-groups interleave over blocks: `for (ib = ix; ib < nb; ib += 4)`

### Our `gemm_q4_k_m_fused_v2` (quant.metal:244)

```
TG threads:  256 (8 simdgroups × 32 threads)
TG output:   8 rows per TG (1 simdgroup per row)
Grid:        ceil(rows / 8) × 1 × 1
```

`simd_id` in 0..7 selects output row. All threads for a row cover 8 elements per block
via `elem = k*32 + simd_lane` (k in 0..7).

## 2. Memory access pattern

### llama.cpp

Activations pre-loaded into **private registers** before the row loop:
```metal
float yl[16];  float yh[16];
device const float * y4 = y + ix * QK_K + 64*iq + 8*ir;
for (short i = 0; i < 8; ++i) {
    yl[i+0] = y4[i+  0];   // first  64-elem half, low chunk
    yl[i+8] = y4[i+ 32];   // first  64-elem half, high chunk
    yh[i+0] = y4[i+128];   // second 64-elem half, low chunk
    yh[i+8] = y4[i+160];   // second 64-elem half, high chunk
}
```
After this load, `yl` and `yh` are never re-read from device memory — the row loop
iterates over these registers only. For `nr0=2` rows, the same `yl`/`yh` are reused,
amortizing the 16+16 device reads over 2 output rows.

### Ours

No pre-loading. Inner loop reads device memory every iteration:
```metal
float xv = x[(uint64_t)b * 256ul + (uint64_t)elem];
```
8 reads per thread per block (uncached hot path). While L1/L2 cache may absorb
repeated reads, register access is 4-10× faster than cache.

## 3. dmin subtraction — sumy trick vs per-element

### llama.cpp

Accumulates partial activation sums before the inner loop:
```metal
float4 sumy = {0.f, 0.f, 0.f, 0.f};
for (short i = 0; i < 8; ++i) {
    yl[i+0] = y4[i+  0]; sumy[0] += yl[i+0];
    yl[i+8] = y4[i+ 32]; sumy[1] += yl[i+8];
    ...
}
```
Then applies the dmin correction once per block after nibble accumulation:
```metal
sumf[row] += d * (acc_nibbles_scaled)
           - dh[1] * (sumy[0]*sc8[2] + sumy[1]*sc8[3] + sumy[2]*sc8[6] + sumy[3]*sc8[7]);
```
Cost: 4 multiplies + 3 adds per block (covers all 256 min subtractions).

### Ours

Per-element dmin subtraction in the hot path:
```metal
float w_val = d * (float)s_byte * (float)nib - dmin * (float)m_byte;
```
`dmin * m_byte` is independent of both the nibble and activation, yet computed
8 times per thread per block = **256 redundant FMAs per row per block**.

Perf impact: for a 7168-col matrix (28 Q4_K blocks), this is 28 × 256 = 7,168
wasted FMAs per row. At 28 blocks × 8 rows per TG = 224 block-rows, the
overhead is ~1.8M FMAs per TG that do no useful arithmetic.

## 4. Packed uint16_t q reads vs byte loads

### llama.cpp

Casts the quantized data pointer to `device const uint16_t*`:
```metal
device const uint16_t * q1 = (device const uint16_t *)x[ib].qs + 16*iq + 4*ir;
// ...
FOR_UNROLL (short i = 0; i < 4; ++i) {
    acc1[0] += yl[2*i+0] * (q1[i] & 0x000F);   // nibble lo of byte 2i
    acc1[1] += yl[2*i+1] * (q1[i] & 0x0F00);   // nibble hi of byte 2i+1 (shifted)
    acc1[2] += yl[2*i+8] * (q1[i] & 0x00F0);   // nibble hi of byte 2i
    acc1[3] += yl[2*i+9] * (q1[i] & 0xF000);   // nibble lo of byte 2i+1 (shifted)
}
```
4 uint16_t reads decode 8 nibbles — 2 device-memory transactions per 8 weight elements.

### Ours

Single-byte loads with nibble extraction:
```metal
uchar q = w_q4[bo + 16ul + (uint64_t)pair * 32ul + (uint64_t)i];
uint nib = upper ? ((uint)(q >> 4) & 0x0Fu) : ((uint)q & 0x0Fu);
```
1 byte read per weight element = 8 device-memory reads per 8 weight elements.
Double the memory transactions for the q data.

## 5. simdgroup_matrix — neither kernel uses it for the GEMV path

llama.cpp uses `simdgroup_matrix` only in its GEMM path (batch >1, matrix × matrix).
For decode (GEMV, batch=1), llama.cpp uses `kernel_mul_mv_q4_K_f32_impl` with plain
`simd_sum`, same as our `gemm_q4_k_m_fused_v2`. Our Wedge H (`gemv_simdgroup_f32`)
does use simdgroup_matrix for the f32-weight GEMV path but not the Q4_K path.

Our Q4_K kernel (`gemm_q4_k_m_fused_v2`) correctly uses `simd_sum` — this part
is structurally sound. The perf gap comes from differences 1-4 above, not from
simdgroup_matrix absence.

## Identified design differences (ranked by estimated impact)

| # | Difference | llama.cpp | Ours | Est. impact |
|---|-----------|-----------|------|-------------|
| 1 | **dmin sumy trick** | 4 FMAs/block for dmin correction | 256 FMAs/block | −15–20% decode time |
| 2 | **Multi-row reuse (nr0=2)** | 2 rows per simdgroup; activations amortized | 1 row per simdgroup | −8–12% |
| 3 | **Register-preloaded activations** | `yl[16]`, `yh[16]` in registers; one load | Device read per element in loop | −5–8% |
| 4 | **uint16_t packed q reads** | 4 reads → 8 nibbles | 8 reads → 8 nibbles | −3–5% |
| 5 | **Inner loop unrolling** | `FOR_UNROLL` pragma on 4-iter loop | No explicit unroll | −1–2% |

Cumulative estimated savings if all 5 are ported: **25–40%** on `gemm_q4_k_m_fused_v2`
decode latency. At current mean_us=349, projected mean_us=210–260 after port.

## Port plan for Wedge K (`gemm_q4_k_m_simdmat`)

New kernel name: `gemm_q4_k_m_simdmat` in `shaders/quant.metal`.
Leave `gemm_q4_k_m_fused_v2` untouched as fallback.

Ordered by implementation sequence (each item depends on previous):

### Step 1 — sumy trick (~20 LoC)
In the inner `k` loop (0..7), accumulate `float sumy = 0.0f` by adding
`x[b*256 + elem]` before the nibble FMA. After the k-loop, apply:
```metal
partial -= dmin * scale_min * sumy;
```
where `scale_min` is extracted once per block alongside `d`/`dmin`.
This removes `dmin * m_byte` from the per-element hot path entirely.

Implementation note: `m_byte` is the same for all 32 elements in a sub-block.
`sumy` = partial sum of activations for those 32 elements (covered by one simdgroup
via `k * 32 + simd_lane`). One `simd_sum(sumy)` per sub-block correction.

### Step 2 — register pre-loading of x (~8 LoC)
Declare `float xl[8]` in thread-private memory. Before the k-loop, load:
```metal
float xl[8];
for (uint k = 0; k < 8u; ++k) {
    xl[k] = x[(uint64_t)b * 256ul + (uint64_t)(k * 32u + simd_lane)];
}
```
Replace `x[...]` reads in the k-loop with `xl[k]`. Keeps activations register-resident
for the full block iteration.

### Step 3 — multi-row (nr0=2) (~25 LoC)
Change geometry: 4 simdgroups per TG (128 threads), each covering 2 rows.
```metal
uint base_row0 = gid * 4u * 2u + simd_id * 2u;
uint base_row1 = base_row0 + 1u;
float partial0 = 0.0f, partial1 = 0.0f;
```
Both rows share the `xl[8]` preload and `sumy`; only the weight fetch changes
(`row_byte_off0` vs `row_byte_off1`). Grid becomes `ceil(rows/8) * 128`.

### Step 4 — uint16_t packed q reads (~10 LoC)
Replace the byte-load + nibble-shift sequence with:
```metal
uint byte_idx = bo + 16ul + pair * 32u + (i & ~1u);  // aligned to 2-byte boundary
device const uint16_t * qp = (device const uint16_t *)(w_q4 + byte_idx);
uint16_t qpair = *qp;
uint nib = (i & 1u) ? ((qpair >> 8) & 0x0Fu) : (qpair & 0x0Fu);  // low byte
// for upper half: shift by 4 from respective nibble
```
This halves device memory reads for the q data path.
Note: alignment guarantee — Q4_K block `qs` field is 128-byte aligned within a 144-byte block.

### Step 5 — explicit unrolling (~5 LoC)
Add `[[unroll(8)]]` attribute to the `k = 0..7` inner loop and `[[unroll(2)]]` to the
row loop in Step 3. Enables the compiler to pipeline the 8 independent k-iterations.
