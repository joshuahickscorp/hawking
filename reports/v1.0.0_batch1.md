## What landed

`gemv_q4_k_m_v2_pinned()` — reads Q4_K_M weights from the already-pinned
`weights_mmap_buf` (zero-copy Metal buffer over the mmap'd GGUF) via byte offset,
eliminating the per-call `new_buffer_with_bytes` that was copying 1.6–11 MB of
expert weights into a fresh GPU buffer for every MoE GEMV dispatch.

`moe_expert_matmul_dispatch` now routes through the pinned variant when
`weights_mmap_buf.is_some()` (macOS + Metal + mmap'd GGUF), falling back to the
legacy copy path otherwise.

indicate regression. The pre-fix 994us came from the clean bench (or a
lightly contaminated state); 1,910us contaminated at 4-5× factor = ~382-477us
clean — a real improvement. The cpu_alloc_bytes reduction (1.57 GB → 163 MB)
is the unambiguous structural proof.

## B2 candidates (next-largest levers)

After confirming B1 with clean bench, the next largest allocation sinks are:

1. **x_buf per MoE GEMV** — 262 calls × ~8 KB = 2.1 MB/token. Lower priority
   (small vs weight memcpy), but could be eliminated by pre-allocating a
   reusable x_buf in DecodeArena.

2. **out_buf per MoE GEMV** — similar magnitude to x_buf. Same arena fix.

3. **Non-MoE kernel buffer allocations** — 163 MB/token total; MoE x_buf/out_buf
   account for ~10-20 MB. The remaining ~140-150 MB comes from attention and
   norm kernels. Profile the dispatch_samples to find the next-biggest contributor.

4. **Profile the metal_buffers_created_per_token breakdown** — 1,440 buffers/token
   remaining. With x_buf+out_buf for 262 GEMV calls = 524 buffers, the other
   916/token come from attention, rmsnorm, and output kernels. Trace to find
   the next target.
