## What landed (1 commit)

### `ebf3249` v2.1.0-T2.1: flip MoE Q4_K schedule to v2t_gu_v2

The v2t_gu_v2 fused gate+up kernel (built in v1.1.1 commit `2902ad9`,
parity-tested via `tests/v1_2_q4k_gu_v2_parity.rs`) was never wired as
the default. The profile sat at `gemm_q4_k_schedule = "per_shape"` which
fell back to `moe_batched_gemm_q4_indexed_v2` — the basic v2 kernel
without threadgroup x-preload or gate+up fusion.

**Why this had been missed:** the v1.1.0 architecture audit's TCB trace
attributed `moe_batched_gemm_q4_indexed = 0.1%` (which is CPU dispatch
time, not GPU compute time). The audit explicitly flagged this
measurement gap, but every plan since (Phase 4, Phase 5, v2.1 sketch,
v2.1 comprehensive plan) inherited the misread and pointed elsewhere.
MoE GEMV is in fact the 60-65% slice; the actual hot kernel was running
the wrong variant.

**Changes:**
- `profiles/deepseek-v2-lite-q4.m3pro18.json`: `gemm_q4_k_schedule` →
  `v2t_gu_v2`. The MoE TCB path now routes routed gate+up through
  `encode_batched_gemv_fused_gu_v2_tcb` (sumy trick + scale preload +
  paired nibble reads + threadgroup x-cache) and the down projection
  through `moe_batched_gemm_q4_indexed_v2t` (threadgroup x-cache).
- `crates/dismantle-core/src/model/deepseek_v2.rs`: guard the standalone
  Q4_K dispatch path so v2t_*-family schedules don't silently fall
  through to scalar `gemv_q4_k_m`. Maps `v2t`, `v2t_gu`, `v2t_gu_v2`,
  `v2t_gu_serial` → `gemv_q4_k_m_v2_pinned` for non-TCB fallback dispatch.

**Validation:**
- `cargo test --release --workspace --lib`: 25/25 PASS
- `cargo test --release -p dismantle-core --test v1_2_q4k_gu_v2_parity`:
  2/2 PASS (small + production 1408x2048)
- Smoke ("The capital of France is"): "Paris." (sensible English)
- TOKENS=24 and TOKENS=64 benches both clear +5% gate by 3×

## What was tried and reverted

### T2.5 — pre-rmsnorm fusion (q_a + kv_a share input)

**Hypothesis:** the fused `rmsnorm_gemv_f16w_attn_pinned` kernel computes
the variance per-threadgroup (1024 + 576 = 1600 redundant variance
calculations across q_a's 1024-row + kv_a's 576-row dispatches per layer
× 27 layers). Pre-computing rmsnorm(x_buf, attn_norm) once into x_norm_buf
and feeding it to plain GEMVs should save the redundant variance work.

**Implementation:** replaced the two `rmsnorm_gemv_f16w_attn_pinned_tcb`
calls with one `rmsnorm_metal_buf_tcb` + two plain `gemv_f16_metal_buf_tcb`
in `encode_attention_phase1_into_tcb`.

**Bench result:** 19.99 dec_tps median (CI [19.55, 20.44], IQR 0.78) vs
T2.1 baseline 20.997. **−4.8% e2e regression.** Below the +5% gate.
Reverted.

**Root cause analysis:** the GPU runs many threadgroups concurrently
across SMs. The 1600 "redundant" variance computations are parallel work
that finishes in roughly the same wall-clock as 1 variance computation
would take. The savings from eliminating that parallel work are
illusory; meanwhile the added rmsnorm dispatch costs real overhead
(pipeline lookup + threadgroup launch + 8KB scratch write).

**Lesson for future sessions:** "redundant" computations on GPU need a
bench gate to validate, not just a static call-count argument. Per-
threadgroup work that runs in parallel across SMs is approximately free.

### T2.6 — v2t_gu_serial schedule

**Hypothesis:** serial route dispatch (one expert at a time instead of
6 in parallel) means each expert's weight slab is one sequential memory
stream. Should eliminate L2 cache thrashing from competing expert reads.

**Implementation:** profile schedule → `v2t_gu_serial`.

**Bench result:** 18.31 dec_tps median (CI [17.85, 18.76]) — **−12.8%**
vs v2t_gu_v2 baseline. Reverted.

**Root cause:** v2t_gu_serial uses the OLDER `moe_batched_gemm_q4_indexed_v2t_gu`
kernel (not v2t_gu_v2). That kernel lacks the sumy trick, scale-preload,
and paired-nibble-read optimizations from v2t_gu_v2. The serialization
loses on-GPU parallelism faster than the L2-thrashing-elimination wins
back. To get the L2 benefit one would need a `v2t_gu_v2_serial` variant
which doesn't exist yet.

## What was investigated and deprioritized

### T2.3 — LM head top-K kernel restoration

The Phase 5B.2 top-K LM head kernel (commit `6b4c95a`, removed `098a5e7`)
adds a top-K extraction pass after the LM head GEMV. **Analysis showed
this only helps non-greedy sampling paths.** Greedy argmax (the current
hot path) computes the full LM head GEMV anyway and picks one max — the
top-K kernel doesn't reduce the LM head's weight-bandwidth read (419 MB
per token of f16 weights). Restoring it would only help temperature/top-K
sampling, which isn't the default benchmark.

Decision: not restored. Document in next-session priorities for sampling
workloads only.

### `moe_schedule = "single-kernel"` fused dispatch

`moe_block_fused_v2lite_dispatch` fuses gate + up + silu + down + accumulate
into one kernel (`moe_block_fused_v2lite_indexed`). However, this path is
**non-TCB** — it returns `Vec<f32>` to the CPU and breaks the single-CB-
per-token optimization that the `forward_token_shared_only_gpu_argmax`
TCB path provides. Adopting it would gain kernel fusion but lose the CB
amortization. Net effect uncertain; would require a TCB-compatible
rewrite of the fused kernel before testing.

Decision: skip for tonight; investigate post-Tier 1 GPU-counter data.
