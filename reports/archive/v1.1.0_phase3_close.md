## Parallelization experiment retro

### Setup
- Manager: Sonnet 4.6 (this session)
- Workers: 7 Haiku subagents via Agent tool (`run_in_background: true`)
- Task class: mechanical `set_bytes` ‚Üí `KernelArgBuffer` refactor per kernel

### What worked

**Haiku can handle mechanical refactors.** All 7 kernels assigned to Haiku
workers eventually produced working code. The refactor pattern was clear
enough (read set_bytes calls, create argbuf with matching layout, set
values, bind at same buffer index) that Haiku could execute it without
architectural judgment.

**Background parallel dispatch is fast.** Spawning 7 workers concurrently
saved real wall time vs sequential Sonnet ‚Äî estimated 4‚Äì5√ó speedup for the
mechanical refactor step (the ‚â§50 LoC-per-kernel edits that Sonnet would
have done sequentially).

**Isolated worktrees work.** W3 ran in a dedicated git worktree branch and
produced the cleanest diff ‚Äî no race contamination. Cherry-pick with manual
corrections was a small overhead.

### What didn't work

**Shared-file race is the primary risk.** All 7 workers wrote to the same
`kernels/mod.rs` file. W5 committed first and captured in-progress disk
writes from W1, W4, W7 that were staged but not yet committed. Result: 5
kernels bundled into one commit (`867b948`) and subsequent commits only
showed shader changes. The git log is technically correct (all changes are
present and attributed) but the audit trail is misleading ‚Äî each commit
doesn't cleanly map to one kernel as intended by CLAUDE.md's single-purpose
rule.

**Haiku uses wrong access syntax.** W3 wrote `tcb.ctx()` (method call) when
the field is public (`tcb.ctx`). This required manual correction before
cherry-pick. Haiku doesn't reliably distinguish pub field from accessor
method when both are plausible patterns in the codebase.

**W5 bundled too aggressively.** W5 read `kernels/mod.rs` AFTER W1/W4/W7
had already written their uncommitted changes to disk, then committed the
whole file. This produced a correct-but-opaque commit. Manager couldn't
detect this until post-hoc diff inspection.

**W3 worktree isolation adds cherry-pick friction.** A worker in an
isolated worktree produces a commit on a detached branch, requiring manager
to cherry-pick and potentially fix conflicts. For 7 workers, if each ran in
its own worktree this would be manageable; if all share one worktree the
race dominates.

### Recommendation for Phase 4+

**Use Sonnet+Haiku for N-similar tasks with STRICT file assignment ‚Äî
one Haiku per file or per isolated worktree branch.**

The race condition is fixable with discipline: each Haiku worker receives
an explicit branch name and does `git checkout -b worker-N` before any
write. Manager cherry-picks all branches after completion. This adds ~15
min overhead but removes the bundling problem entirely.

**Do NOT use this pattern for:**
- Tasks where all workers write the same shared file (today's problem)
- Architectural pieces requiring Sonnet judgment (ICB, KV fusion, attention math)
- Tasks with non-obvious API patterns (Haiku guesses syntax incorrectly)

**Phase 4 sub-tasks that fit:** if n-gram spec logic splits into N
shape-specific or model-specific sub-implementations, Haiku can handle
each independently as long as each owns a distinct file.

Phase 4 decision: n-gram speculation + batched verify is one architectural
piece (not N similar tasks). Run as solo Sonnet. Parallelization may apply
to sub-tasks within Phase 4 once the architecture is fixed.
