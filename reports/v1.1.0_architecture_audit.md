# v1.1.0 architecture audit — where the V2-Lite 60ms/token actually goes

Date: 2026-05-27
Anchor: V2-Lite Q4_K_M, M3 Pro 18GB, profile shader_hash matched.
Measurement: `DISMANTLE_TRACE_DISPATCH=1 dismantle bench --max-new-tokens 32`

## Headline numbers

- Decode: 1932 ms / 32 tokens = **60.4 ms/token = 16.56 dec_tps**
- Visible dispatch overhead: **78.3 ms total / 32 tokens = 2.4 ms/token = 4.1% of decode**
- Inside-TCB compute (untracked): **~58 ms/token = 95.9% of decode**

The TokenCommandBuffer single-CB-per-token (Codex's v0.5.0+ work) made dispatch
overhead negligible. The remaining 95.9% of decode time is actual GPU compute
inside the TCB. **The current bottleneck is GPU bandwidth + compute, not
dispatch frequency.**

## Per-kernel cost at production shapes

Measured via `dismantle bench-kernel` on the live binary.

| kernel | shape | mean (μs) | role |
|---|---|---:|---|
| gemv_q4_k_m_v2_pinned_tcb | 1408×2048 | 441 | MoE gate+up per active expert |
| gemv_q4_k_m_v2_pinned_tcb | 1024×2048 | 299 | q_a_proj |
| gemv_q4_k_m_v2_pinned_tcb | 4096×1024 | 383 | q_b_proj |
| gemv_q4_k_m_v2_pinned_tcb | 2048×4096 | 359 | o_proj |
| gemv_f16_metal_pinned | 102400×2048 | 3,618 | LM head |

**Important:** these per-call costs are NOT additive when kernels are in TCB.
The GPU runs many concurrently across SMs. Naive sum-of-isolated estimates
~234ms/token but actual inside-TCB is ~58ms. The GPU is achieving real parallelism.

## What this means for the v1.1.0 roadmap

### Phase 3 (Metal ICB) target is optimistic

Roadmap projected Phase 3 at +5-10% e2e from "eliminating per-token CB build
overhead." But our visible dispatch overhead is **already 4.1%**. ICB ceiling
is +4.1% e2e (zero-out dispatch overhead entirely), probably less in practice
(some sync points are unavoidable). **Realistic Phase 3 e2e win: +1-3%.**

Phase 3 still has VALUE (argument-buffer refactor unblocks future work, ICB
infrastructure is needed for batched verify), but treating it as a perf
phase was wrong — it's primarily an ENABLER phase.

### Phase 4 (n-gram spec + batched verify) is the actual unlock

N-gram speculative decoding is already in tree (commit `2b5dbd2`, v1.0.5).
With serial verify (current state), α=0.4 acceptance gives ~0% e2e win
because each verify costs a full forward pass.

With batched verify (Phase 4 deliverable), α=0.4 acceptance gives
**+30-40% e2e on natural text**, +60-80% on repetitive text.

This is the highest-leverage work in the v1.1.0 roadmap. **Phase 4 should
be prioritized over further Phase 3 polish.**

### What's blocking Phase 4

Phase 4 needs:
1. Real `forward_tokens_batched` (currently a sequential loop over `forward_token`)
2. Batched MLA decode kernel (`mla_decode_kernel_batched_slots` exists from
   Codex's work but isn't fully wired)
3. Batched MoE expert dispatch (NEW kernel needed)
4. N-gram spec verify path that uses batched forward

Most of this work IS infrastructure that Phase 3 was supposed to lay down.
Specifically, argument-buffer refactor (Phase 3 sub-task) is needed for ICB
which is needed for low-overhead batched dispatches.

## Findings: where we can't see

The current `DISMANTLE_TRACE_DISPATCH` mechanism only records `dispatch_batch`
and `dispatch_threads` calls that go through `MetalContext` directly. Calls
through `TokenCommandBuffer::dispatch_threads` are NOT recorded individually
because they don't get their own commit.

**This is a measurement gap.** We cannot currently determine:
- Which specific kernel takes the most time inside the TCB
- Whether MoE expert GEMVs or attention projections dominate
- Whether kernel fusion (e.g., rmsnorm+QKV fusion) would yield real wins

## Recommendations

### Short term (next 1-2 sessions)
1. **Dispatch Phase 3 as planned, but with checkpoint:** if Phase 3 lands ≤+3%
   e2e (within the audit's projected range), proceed to Phase 4 immediately
   without further Phase 3 polish.
2. **Add TCB-internal instrumentation** as part of Phase 3's argument-buffer
   refactor (it's a natural pairing — both touch Metal command encoding).

### Medium term (Phase 4)
1. **Phase 4 is now the priority phase.** Batched verify with the existing
   n-gram spec is the largest single e2e win available.
2. Phase 5 (KV-aware fusion + fp16 KV) becomes more important too — it
   reduces bandwidth which IS the actual bottleneck per this audit.

### Long term (Phase 6 moonshot)
The audit doesn't change Phase 6's calculus much — the moonshot phases
(ANE, AMX, persistent kernels, INT4 activations) all attack genuine
bandwidth/compute ceilings.

## Conclusion

**The per-kernel optimization era is mostly done.** v0.5.0 → v1.0.5
delivered the architectural rewrite (single-CB-per-token, pinned mmap,
on-GPU top-k). v1.1.0 Phase 1 + 2 attempted per-kernel improvements (Q4_K
port, MPSGraph LM head) but they don't compound at e2e because the bottleneck
shifted from dispatch to compute inside the TCB.

**Future phases should attack: bandwidth (fp16 activations + KV), compute
amortization (batched verify with spec), and parallel compute units (Phase 6
ANE/AMX).** Per-kernel speedups will keep landing as opt-in infrastructure
without moving e2e materially.

**The realistic v1.1.0 trajectory revised:**

| phase | original target | revised target | rationale |
|---|---:|---:|---|
| 1 | 26-30 | 16-17 (landed) | per-kernel wins didn't compound |
| 2 | 30-35 | 16-17 (landed) | MPSGraph hit per-call overhead floor |
| 3 | 32-38 | 17-18 | ICB targets the small remaining 4% dispatch overhead |
| 4 | 42-50 | **30-40** | batched verify on n-gram spec is the real unlock |
| 5 | 52-60 | 40-50 | bandwidth reduction + kernel fusion |

**60 dec_tps remains plausible AT Phase 5, but most of the gap closes at
Phase 4 not Phase 3.** Phase 3 should be treated as enabler (lay down ICB +
argument buffers + TCB instrumentation), Phase 4 as the perf inflection point.
