## Diagnostic path

The v1.0.4 prompt anticipated **RoPE theta** as the most likely first bug.
Diagnostics proved that wrong — RoPE was already configured correctly with
theta=1000000. The actual first divergence was **tokenizer semantics**:

- Initial state: dismantle's GGUF tokenizer fell back to GPT-2 byte-level
  for any vocab tagged `tokenizer.ggml.model = "llama"`. Mixtral uses
  SentencePiece-style ▁-prefixed pieces, so common prompts became byte
  fallback IDs.
- Comparison vs llama.cpp on `"Once upon a time"`:
  - dismantle (broken): `[1418, 358, 582, 266, 264, 261, 545]`
    (" On" "ce" " up" "on" " a" " t" "ime")
  - llama.cpp (correct): `[BOS, 5713, 3714, 264, 727]`
    (BOS " Once" " upon" " a" " time")
- Garbage prompt → garbage attention context → flat logits → repeated
  newline output.

## Fix progression

**Fix attempt 1** (`c0b276f`): switched to HF tokenizers' Unigram model
for SPM vocabs. Output went from `\n\n\n\n...` to `, importing a` — coherent
fragments but still wrong tokenization (16 tokens for "Once upon a time"
where llama.cpp produces 5).

**Fix attempt 2** (`e1ee90c`): ported llama.cpp's actual `tokenizer.cpp`
greedy score-merge algorithm — symbol-linked-list + binary-heap-of-bigrams
with byte-fallback via `<0xXX>` tokens. Added BOS/EOS handling driven by
GGUF's `tokenizer.ggml.add_bos_token` metadata. This produced exact
tokenization parity with llama.cpp and unblocked coherent generation.
