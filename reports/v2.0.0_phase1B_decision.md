# Sub-phase 1B decision report

**Date:** 2026-05-13
**Decision: NO — spec decode is structurally dead in dismantle.**
**Recommendation: pivot to Path B (ship at honest framing).**

## What was measured

Three workloads at TOKENS=64, single trial each, via `dismantle generate` CLI
(bypassing the bench-server bug from Sub-phase 1A which would have masked
the architectural finding).

| workload | spec mode | dec_tps | decode_ms | accepted/rejected | α |
|---|---|---:|---:|---|---:|
| natural English | OFF | 17.67 | 3622 | 0/0 | — |
| natural English | NGRAM | 16.64 | 3845 | 0/4 | 0.00 |
| code (Python) | OFF | 17.64 | 1927 | 0/0 | — |
| code (Python) | NGRAM | 10.10 | 3365 | 3/25 | 0.11 |
| repetitive (numbers) | OFF | 17.49 | 3659 | 0/0 | — |
| repetitive (numbers) | NGRAM | 17.41 | 3675 | 51/0 | 1.00 |

## The architectural finding

**Spec NGRAM at 100% acceptance gives -0.5% e2e.** That's the smoking gun.
In a properly batched spec decode, α=1.0 should yield ~K× speedup (K=4 →
+300%). Instead: flat.

### Root cause

`forward_tokens_batched_tcb` in `crates/dismantle-core/src/model/deepseek_v2.rs`
line ~2240 is documented:

> "Processes K tokens **sequentially** within one Metal command buffer."

The "batched verify" is K serial forwards encoded into one CB. Savings:
only the CB-commit overhead (~3% per the Phase 4 trace's 4.1% dispatch
slice). Compute is unchanged: K verify tokens cost K full forward passes
of GPU compute.

### What's needed for spec to actually work

True K-parallel spec verify requires:
- Batched MLA decode where K queries process in parallel across SMs
  (Phase 4A built `mla_decode_metal_batched_slots` but it was never wired
  into `forward_tokens_batched`)
- Batched MoE expert GEMV where K tokens' active experts dispatch in
  parallel (NEW kernel work — never built)
- Correct K-aware DecodeArena buffers (Phase 5A added this scaffold but
  the per-layer kernels still process serially)

Estimated work to actually deliver batched verify: 3-4 weeks. No guarantee
it would beat the bandwidth/dispatch ceiling.

## Why this changes the v2.0 plan

The v1.1.0 architecture audit (commit `3f124f9`) said:

> "Phase 4 (n-gram spec + batched verify) is the actual perf unlock the
> v1.1.0 roadmap pointed to."

That projection assumed batched verify was real K-parallel compute. It
isn't. Phase 4D wired n-gram spec correctly (exact-mode invariant proven)
but the verify path it relies on was never made truly parallel.

**Sub-phases 1B (lookahead/tree spec) and indirectly all spec-decode work
in v2.0 Phase 1 inherit this same limitation.** Lookahead and tree spec
both depend on the same verify path. Fixing the verify is a prerequisite
to ANY spec decode delivering real wins.

## Combined data: where dismantle actually stands

Across all measurements (1A + 1B):

| condition | dec_tps | source |
|---|---:|---|
| Default V2-Lite | 17.7 | 1A baseline |
| Combined opt-in (no spec) | 17.9 | 1A combined opt-in |
| Spec ngram, natural | 16.6 | 1B (-5.8% from overhead) |
| Spec ngram, code | 10.1 | 1B (-43% from rejected drafts) |
| Spec ngram, repetitive @α=1 | 17.4 | 1B (-0.5% — best case is zero benefit) |

**No spec configuration delivers ≥+5% e2e on any workload.** The
infrastructure is correct (parity proven) but doesn't translate to perf.

The combined opt-in (fp16 KV + fp16 x_norm) compounds to +0.86%. Within
bench noise.

## Recommendation: Path B

Stop the perf push. Ship at honest framing.

### What to ship

1. **Default V2-Lite: ~17 dec_tps** (real measurement, with documented CIs)
2. **Mixtral 8×7B Q3_K_M on 18GB Mac** (genuinely unique, unlocked by
   Sub-phase 1A's Q3_K kernel work)
3. **Pure-Rust + Metal MoE engine** (positioning differentiator)
4. **Reproducible kernel autotune + open methodology** (bench-server,
   bench-kernel, statistical CIs in tree)
5. **Single binary, no Python** (operational simplicity)

### What NOT to ship as a perf claim

- Don't claim spec decode wins anything — it's net negative on every
  workload tested
- Don't claim fp16 KV/x_norm helps — combined opt-in is +0.86%
- Don't promise 60 dec_tps — it's not achievable on this architecture
  in any reasonable timeline

### Phase 2 cleanup (~1 week)

Now that 1B confirms spec doesn't help and combined opt-in is flat,
remove the dead code:

- `--speculate ngram` mode (and supporting `speculate/ngram.rs`,
  `speculate/shared.rs` for ExactShared) — never delivered, removes ~600 LoC
- `kv_cache_dtype="f16"` profile path + f16 MLA kernels — opt-in only,
  +0.86% combined. Remove the kernels + dispatchers, keep f32 default
- `x_norm_dtype="f16"` profile path + kernels — same rationale
- LM head top-K kernel from Sub-phase 5B.2 — built but never wired into
  generate path; pure dead code
- Phase 1B llama_port + per-shape autotune kernel from v1.1.0 — opt-in
  only, never flipped default
- Multiple `_tcb` variants and parity tests for above

Estimated removal: ~2,000-3,000 LoC across kernels/mod.rs, shaders, tests.
Result: simpler codebase, easier to maintain, focused on what works.

### Launch prep (~1 week)

- README rewrite around the actual story (Mixtral demo, pure Rust, methodology)
- Mixtral docs polish — fetch-mixtral.sh, recommended commands, expected speed
- Bench results page showing default V2-Lite numbers + Mixtral demo
- Release tag plan
- r/LocalLLaMA post draft material

### What to do with the moonshot work

Don't dispatch 1C (ANE), 1D (AMX), 1E (ICB), 1F (INT4), 1G (sparse).
Each is multi-week with uncertain payoff. The combined optimistic case
was 28-32 dec_tps — meaningful but not 60 — and the realistic case is
22-25. If we want to keep pushing perf post-launch, do it AFTER getting
real user feedback from the v2.0 release. User input shapes which
optimizations matter for actual workloads.

### If we eventually want spec decode to work

The right architectural project: build true K-parallel batched forward
(real K-axis parallelism in MLA, MoE GEMV, rmsnorm). That's a v2.x
multi-month project. Park it as future work; don't gate v2.0 launch on it.
