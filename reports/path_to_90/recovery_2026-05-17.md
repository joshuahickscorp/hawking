# Path-to-90 recovery — 2026-05-17

Recovery from an orphaned worktree situation and a latent data-prep bug.
Single source of truth for what was lost, what was changed, and what
replaces what going forward. Pairs with the commit that updates
`NEXT_SESSIONS_BRIEF.md`.

## What happened

The previous session left two processes running in
`.claude/worktrees/dreamy-golick-d54ff8/`:

- PID 99052 — `dismantle capture-hidden` writing `shard_000.bin`
- PID 99053 — `tools/training/pipeline_loop.sh` polling for completion

The worktree directory was subsequently removed from the filesystem
(`git worktree list` no longer showed it) **while both processes were
still running**. Standard Unix unlink-while-open kept the file descriptors
alive — `lsof` confirmed PID 99052 still had write handles on
`shard_000.bin` (inode `92342450`) and `shard_000.log` — but the data was
unreachable via any path. The .bin had grown to ~216 MB before recovery
and would have been lost the moment the process exited.

The `pipeline_loop.sh` was polling marker files in a directory that no
longer existed; every `[[ -f ... ]]` check returned false. It would have
spun forever, never firing S1..S10.

## Decision

Restart cleanly. Option-A "lldb fd-recovery" was rejected for three
reasons:

1. The orphan shard's input file (`tests/data/ultrachat_100k_v2_dialogue.jsonl`)
   was also lost with the worktree, so the orphan data could not be
   validated for provenance even if recovered.
2. The 216 MB of in-flight capture was <1% of the planned 100K-sample run.
   Trading 1% of data for a clean, defensible audit trail is the right
   call for a portfolio project.
3. The CLAUDE.md evidence-rule rigor favors clean lineage over best-effort
   recovery; "shard recovered from an orphaned inode via debugger" is
   exactly the footnote that would undermine downstream benchmarks.

Both processes were SIGTERM'd cleanly. The worktree was recreated via
`git worktree add`. The dismantle binary was rebuilt in the recreated
worktree (1m; symbols current with `claude/dreamy-golick-d54ff8` HEAD).
The model file (`models/deepseek-v2-lite-q4.gguf`) was symlinked from
the main repo's `models/` directory rather than re-downloaded.

## Latent bug found and fixed

The committed pipeline scripts (`tools/training/advance_pipeline.sh:335`
and `tools/training/chain_capture.sh:141`) both ran the UltraChat prep
step with `--max-chars 2000`. UltraChat's `train_sft` split contains
multi-turn dialogues; `capture_hidden.py prep` concatenates all turns
into one text with "User:"/"Assistant:" prefixes. Concatenated dialogues
are routinely 3,000-15,000 chars — only ~6% pass the 2,000-char filter,
yielding 12K samples instead of the requested 100K.

This bug had not surfaced before because the previous session manually
generated `ultrachat_100k_v2_dialogue.jsonl` outside the pipeline (with
different filter args, presumably), and the script path was never
exercised. Any future re-run of the committed pipeline would have hit
this and halted at S9.

Both call sites are now `--max-chars 20000`. Re-running prep against the
real HF dataset produces 100K samples; deduping against the existing
55K union yields a 155K-sample `ultrachat_100k_union.jsonl`. The main
capture will be capped to 100K via `--max-samples` to stay on the
brief's ROI plan.

## Brief audit results

A separate audit pass on `NEXT_SESSIONS_BRIEF.md` (463 lines) surfaced
additional drift. Severity-tagged summary; full notes are in the audit
agent's run output (consumed by the commit message). Blockers:

  `ultrachat_100k_v2_dialogue.jsonl`. Replaced with `ultrachat_100k_union.jsonl`.

  `shard_v2_partial_dcap1.bin`) and a checkpoint path

- **Phase 2 prompt** describes `encode_layer(li, tcb)` as if it were a
  method call. In `crates/dismantle-core/src/model/deepseek_v2.rs:2785`
  it is a closure `|tcb: &mut TokenCommandBuffer| -> Result<bool>` that
  captures `li` from the enclosing for-loop. Literal copy of the brief's
  pseudocode would not compile. Also: the loop body has two `encode_layer`
  invocation sites (single-TCB at L2867, per-layer-fallback at L2871) —
  the Phase 2 implementer must patch both, or scope the change to the
  single-TCB branch only.
- **Phase 2 sanity test** (§14) compares the capture output to
  `forward_token_with_hidden_for_test`, but the existing trait method
  returns POST-final-rmsnorm hidden state while the proposed capture
  blit produces PRE-final-norm. The parity test as written would always
  fail. Either apply final_norm to the capture before comparing, or
  expose a pre-norm intermediate via a new trait method for testing.

## Going forward

The committed pipeline (`chain_capture.sh` → `advance_pipeline.sh`) is
the reproducible entry point. The brief's manual-nohup commands and the
hand-built `*_v2_dialogue` input file were a one-session off-script
experiment that drifted from the committed code. Future hauls should run
the committed scripts unmodified. If a script needs to change, the
change goes into the script and a commit, not into a brief.

The previous session's pattern of committing meta.json + log without the
corresponding .bin (which is gitignored as too large) is the structural
weakness that made this recovery necessary. The .bin's existence is not
visible from git; a worktree deletion silently loses it. A small
improvement worth landing in a follow-up: have `capture-hidden` write a
`shard_000.bin.sha256` sidecar that *is* committed, so the audit trail
can detect "the binary that produced this meta is gone" before downstream
training tries to consume it.
