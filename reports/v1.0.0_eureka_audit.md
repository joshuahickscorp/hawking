## TL;DR

**12 wedges have moved e2e from ~2.21 → ~2.30 dec_tps clean.** That's
4% over 6 weeks of focused work. The user is correct: we're not making
leaps. This document explains why, and proposes the three architectural
pivots that could realistically reach 20 dec_tps.

## Why optimization isn't translating

### Eureka 1: 36% of decode is `dispatch_batch` — that's pure overhead

`dispatch_batch` wraps `cmd.commit() + cmd.wait_until_completed()`. Its
3827us mean × 45 calls/token = **172ms of pure GPU submit + sync wait**
per token. The kernels INSIDE these commits do ~98ms of MoE work + ~58ms
of other work, totaling ~156ms of actual compute. **The other 16ms in
dispatch_batch is pure scheduler overhead.**

Worse: those 45 dispatch_batches don't even cover all kernels. Many
kernels go through `dispatch_threads` directly (no batch wrapper). Total
commit_and_wait calls per token: ~400 (from `dispatch_commits_per_token`
trace metric).

llama.cpp Metal probably runs ~10-30 commit_and_wait per token by holding
one CB across many kernels. **That's 5-15× fewer sync points alone.**

### Eureka 2: MoE GEMV at 413us mean is 5-8× off llama.cpp

llama.cpp's `mul_mv_q4_K_f32` runs at ~50-80us mean on this hardware.
Our `gemm_q4_k_m_fused_v2` is 413us. That's not because their kernel
is faster per FLOP — it's because their kernel **doesn't suffer per-call
launch overhead** the way ours does. They use simdgroup_matrix tiles
that maximize work-per-launch.

Wedge K iteration 2 got us from 994us → 262us per call (kernel-level
micro-bench). But in the actual decode trace, the kernel still measures
413us mean — the additional 150us is dispatch/launch overhead that
DOESN'T show in synthetic micro-benches. **The real bottleneck is dispatch
frequency, not kernel quality.**

### Eureka 3: We've been doing the wrong thing for 6 weeks

Almost every wedge (A, B, C, D, E, F, G, H, I, K, L, M) has been a
micro-optimization. Wedges that improved per-kernel speed (K: 1.78×,
H: faster GEMV, L: theoretically faster attn): **none flipped to
default**, none e2e-validated. Wedges that consolidated dispatch
(B, C, D, M): structural metrics moved, e2e moved 4-7%.

The pattern: **per-kernel improvements aren't translating because the
dominant cost isn't kernel time — it's dispatch frequency.**

flip L flash default, add real fp16 throughout, finish MPSGraph for

| change | save ms/token | running total |

The 121ms CPU/sync gap and 172ms dispatch_batch overhead are the unaddressable
floor without architectural pivots.

## The 3 architectural pivots that would actually get us to 20+

### Pivot A: Single command buffer per token (commit-collapse)

**Replace 400+ commits/token with 1-5.** Hold one MTLCommandBuffer through
the entire forward pass. Encode every kernel as a separate compute encoder
into the same CB. Commit once at the end of the forward pass.

The hard parts:
- Metal hazard tracking must work across all encoders. Different encoders
  reading/writing the same buffer get auto-barriers — verified safe.
- Sync barriers (gate logits → CPU top-k) currently force commit. Solution:
  pre-compute all expert routing on GPU using a top-k kernel, eliminating
  the CPU readback. Or: tolerate a small number of commits (one per
  layer's gate readback ≈ 27/token).

Realistic outcomes:
- Optimistic (1 commit/token): 480 → 100ms/token = **10 dec_tps** (~5×)
- Conservative (27 commits/token, one per layer): 480 → 280ms/token = 3.6 dec_tps (~1.5×)

**This is the single highest-leverage change available.** Wedge M attempted
this scope (the prompt called for "≤35 commits/token") but only consolidated
4 of 18 sites because the work was scoped to high-level TCB sites, not
the low-level dispatch_batch wrappers inside kernel dispatchers.

### Pivot B: Full MoE kernel fusion (one kernel = whole MoE block per layer)

**Replace 23 calls/layer (gate + 6×3 expert + shared) with 1 call/layer.**
A single kernel that:
1. Reads gate logits
2. Picks top-k experts (on-GPU sort)
3. Dequant + matmul for each picked expert in parallel work groups
4. Accumulates weighted result
5. Writes one output buffer

llama.cpp's `mul_mv_q4_K_f32` doesn't do this — they still dispatch per-expert.
But MLX has explored fused MoE kernels. We previously tried this
(`moe_block_fused_v2lite_indexed_metal`) but it was ~200× slower in
testing — possibly an implementation bug, not a fundamental limitation.

Realistic outcomes:
- If kernel achieves 50% bandwidth: 98ms → 25-35ms = 480 → 410-420ms/token
  = 2.4 dec_tps (~+5%)
- Combined with Pivot A: 100ms → 30-40ms drop in MoE = 70ms/token = **14 dec_tps**

### Pivot C: MPSGraph for non-MoE matmuls (faithful, this time)

**Wedge H attempted this with the simdgroup_gemv fallback.** Real MPSGraph
integration could compile the entire `attention()` block + `lm_head` as
ONE graph that submits as a single Metal kernel from MPSGraph's compiled
result.

The hard parts:
- objc2 v0.5 vs v0.6 FFI conflict (documented in v0.B.1 HALT)
- Workspace pinned to v0.5 for metal-rs compatibility
- Solution: write minimal FFI shim in our own crate using objc2 v0.5 raw
  msg_send! — bypass the upstream `objc2-metal-performance-shaders-graph`
  crate that requires v0.6

Realistic outcomes alone: 480 → 420ms (+15%). Combined with A+B: ~50ms.

## The path to 20 dec_tps

Sequential pivots (each builds on prior):

| stage | description | LoC | risk | end dec_tps |
|---|---|---:|---|---:|
| now | post-Wedge-M | — | — | 2.30 |
| **N** | Pivot A — single-CB-per-token (real this time) | ~1500 | high | 7-10 |
| **O** | Pivot B — fused MoE kernel | ~1000 | high | 12-16 |
| **P** | Pivot C — MPSGraph FFI + LM head/attn projections | ~800 | medium | 18-25 |
| **Q** | (if needed) Pivot D — fp16 activation default + sampling fixes | ~500 | low | 20-30 |

**Realistic v1.1.0 ship (4-7 weeks of focused work):** 18-25 dec_tps.
**That meets the 20 dec_tps minimum.**

Risk factors that could cap us below 20:
1. Pivot A might hit a Metal hazard ceiling (some kernels can't share CB
   safely). Floor: ~150 commits/token → 5-7 dec_tps.
2. Pivot B might fail due to GPU sort not being efficient at this size.
3. Pivot C MPSGraph FFI might be too painful to make ergonomic.

## Recommendation: Wedge N now

Don't ship at 2.30 even as v0.6.0. Instead, do Pivot A as Wedge N —
the highest-leverage single architectural change. If it lands, we're at
7-10 clean and a v0.7.0 ship makes sense. Then Pivot B as Wedge O. Then
Pivot C as Wedge P. Each STUCK protocol has a fallback (revert and ship
at the previous pivot's number).

The 20 dec_tps target is genuinely achievable but it requires committing
to the architectural pivot path, not more micro-wedges. The runway is
4-7 weeks. The user said "minimum 20" — this is the realistic plan.
