# v1.2.0 Phase 3 BLOCKED

Phase 3 investigated persistent compiled command buffers via Metal indirect
command buffers.

Findings:
- The `metal` crate exposes `IndirectCommandBufferDescriptor`,
  `IndirectCommandBuffer`, and `IndirectComputeCommand`.
- `IndirectComputeCommandRef` supports pipeline state, kernel buffers,
  threadgroup memory, barriers, and dispatch.
- It does not expose an equivalent to the current `set_bytes` pattern used
  throughout the decode path for scalar kernel arguments.
- The current TCB path encodes many per-dispatch scalar constants with
  `set_bytes`; moving to ICB safely would require first converting those
  scalar args to stable argument/constant buffers across the full forward
  pass.

Gate:
- Required: parity plus coexist median >= 29 dec_tps.
- Result: not attempted because the needed ICB representation is not a thin
  wrapper around the current TCB encoder.

Decision:
- No code changes retained.
- No Phase 3 commit.
- Phase 4+ can continue on the existing single-TokenCommandBuffer path.

## Session 2 assessment (2026-05-12)

Structural blocker confirmed unchanged. The set_bytes refactor is a multi-day
undertaking across dozens of dispatch sites. Not a 5-iteration fix. SKIP.
