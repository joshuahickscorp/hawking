mode that records per-dispatch GPU duration in the REAL production
single-CB pipeline via Apple's boundary-mode counter sampling

## What was attempted and rejected

**L4 — kv_b_proj f16 pin + `mla_decode_kernel_f16w`**: built clean,
parity bit-identical, **+0.94% e2e** (below ~5% coexist noise floor).
Reverted. Lesson: weight-bandwidth opt only pays when read set exceeds
cache. kv_b_proj is 32 KB/call per head — fits cache, halving DRAM
doesn't matter.

**L8 — pin dense FFN gate/up/down as f16 + route to `gemv_f16_simdmat_tcb`**:
built clean, parity bit-identical at 8 tokens. ProdCbGpu trace
confirmed `gemv_f32_attn` → `gemv_f16_simdmat` migration. Bench A/B
**+0.83% median / +0.13% trimmed_mean** — below noise. Reverted.
Lesson: `gemv_f16_simdmat` doesn't deliver on dense FFN shape
(10944×2048) — per-call us actually rose slightly (919 us vs 904 us).
Simdmat tiling is tuned for attention projections, not the much-taller
dense FFN matrices.

**LM head simdmat (T2.8 revisit)**: profile flip to
`lm_head_schedule = metal-simdgroup-matrix-argmax`. Greedy decode
diverged at token 1 (model output `\nThe...` instead of ` Paris.\n\nThe...`).
T2.8 was reverted for the same reason — fp16 accumulation in
`simdgroup_matrix` flips argmax on close logits. Cannot ship for
greedy-correct users.

**L3 — Q4_K schedule autotune sweep**: ran `autotune_sweep.sh` over
`{v2t_gu_v2, v2t_gu, v2t_gu_serial, v2t}`. Result: **v2t_gu_v2 is the
best** (25.6 dec_tps vs 24.7 / 22.3 / 23.6). The shipped default is
already optimal among current alternatives. No win available without
adding a new schedule variant.

Raw sweep: `bench_results/autotune_sweep_20260514-154244.jsonl`.
