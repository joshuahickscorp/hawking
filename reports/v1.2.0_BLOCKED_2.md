## Attempt 2 (this session): sumy + paired nibbles in v2t_gu_v2

Applied three improvements to `moe_batched_gemm_q4_indexed_v2t_gu` while keeping
the indexed+fused dispatch path intact:

1. **Scale preloading**: sg[8]/mg[8] extracted once per block (both gate+up).
2. **Sumy trick**: `sumy[k] = simd_sum(xl[k])` per sub-block; correction accumulated
   as `total_corr += dm[k] * sumy[k]` (scalar). Eliminates 16 MADs/thread/block.
3. **Paired nibble reads**: pi-loop (4 iters) instead of k-loop (8 iters). One byte
   covers two consecutive sub-blocks.

## Root Cause

The MoE Q4_K_M GEMV is **memory-latency-bound** at decode batch-size=1:

1. Sumy trick removes compute (correction MADs) but not bandwidth — the kernel
   stalls on weight reads, not ALU throughput.
2. Paired nibble reads don't reduce total bytes read (32 lanes × 4 bytes = 128 bytes
   per simd group per block, same as before with coalesced access).
3. Per-expert weight reads (gate ~1.6 MB + up ~1.6 MB = 3.2 MB/expert) exceed L2
   capacity (~6-8 MB), causing 6 simultaneous expert reads to compete for bandwidth.
4. +2.1% improvement is within noise; the kernel is bandwidth-latency-bound and
   compute savings don't translate to wall-clock improvement.

Gap to gate: need +25%, achieved +2.1%. Even a theoretically perfect gate+up kernel
(0 µs) would only give ~+25% total (gate+up ≈ 50% of token time); we're nowhere near
that. The +26 dec_tps gate was predicated on a different baseline (Phase 1 landing at
21+ dec_tps), but since Phase 1 also STUCK, the gap is wider.
