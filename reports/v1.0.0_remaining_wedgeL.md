## Design

**Why flash attention?**

The existing `mla_decode_kernel` allocates `scores[seq_len]` in threadgroup memory (TG shmem). At seq_len=4096 with f32, that is 16KB per TG. Apple M2 has ~32KB TG shmem per shader core, leaving room for ~1 TG per core. This serializes heads into 13 waves for n_heads=128.

`flash_attn_decode_kernel` replaces the materialized score array with a `scores_tile[FLASH_TG=128]` slice (512 bytes) and maintains running online softmax state in `state[8]` (32 bytes). Total TG shmem drops from ~20KB to ~4.5KB, enabling ~6 TGs per core — up to 6× better occupancy and roughly 2 waves for n_heads=128.

**Kernel structure (`shaders/attn.metal`):**
- FLASH_TG=128 threads / TG; FLASH_NSG=4 simdgroups
- Grid: (n_heads × 128, 1, 1); each TG handles one attention head
- Shmem slots:
  - slot 0: `q_nope_proj[kv_lora_rank]` — Phase 0 output, shared across tiles
  - slot 1: `acc[kv_lora_rank]` — running weighted sum (unnormalized)
  - slot 2: `scores_tile[128]` — current tile scores (512 bytes fixed)
  - slot 3: `state[8]` = {m_running, l_running, correction, m_tile, simd[0..3]_max}
- Phase 0: q_nope_proj = w_uk^T × q_nope (parallel across r, same as mla_decode_kernel)
- Flash loop (n_tiles = ceil(seq_len / 128)):
  1. Each thread scores one token in the tile (dot product over kv_lora_rank + rope terms)
  2. `simd_max` per simdgroup → state[4+simd_id]; thread 0 reduces → m_new
  3. Thread 0: online update m, l, correction → state[0..3]
  4. All threads: `acc[r] = acc[r]*correction + Σ_ti exp(s[ti]-m_new)*c_kv[ti,r]`
- Normalize: acc /= l_running
- Phase 4: out = w_uv × acc (same as mla_decode_kernel)

**Routing (`deepseek_v2.rs::mla_decode`):**
- `profile.selected.attn_block_schedule == "flash"` → `flash_attn_decode_metal`
- Default path unchanged (`mla_decode_metal` when schedule is "mla" or unset)
- `KernelVariant.attn_block_schedule` field added to `profile.rs` with serde default "mla"

**Dispatcher (`kernels/mod.rs::flash_attn_decode_metal`):**
- FLASH_TG=128 constant; grid=(n_heads × 128, 1, 1); TG=(128,1,1)
- Sets shmem slots 0-3 as above
- Same buffer/bytes layout as `mla_decode_metal` (buffers 0-11)
