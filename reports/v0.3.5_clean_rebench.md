Quiet-machine gate. Two additional issues discovered during execution

### Issue 1 — Structural bench contamination from Claude executor session

All bench trials run under this executor session produce ~0.39–0.44 dec_tps,
consistent with the v0.3.4 contaminated runs (0.42–0.44 when SpotlightKnowledged
was at ~99% CPU). The expected clean value is ~1.748 tps (v0.3.3 baseline).

The contamination source is the active Claude Code executor session itself:

After prior bench (pid 36834) exited, non-bench state settled to:
```

```

No Spotlight/mds/mdworker processes present. The 40–50% CPU load is
inherent to the active Claude executor session (UI rendering + API
traffic). On an M3 Pro with unified memory, the Claude app's Metal
rendering (WindowServer) and the bench's Metal compute commands share
the same GPU memory bandwidth. Running the bench FROM a Claude
session means the GPU is never idle.

Simd trial 1 result (discarded — contaminated):
- completion_tokens: 57
- decode_ms: 146,948ms
- decode_tps: 0.388 (vs expected ~1.748; factor-4.5 slowdown)

Cross-check: the previous session's bench (pid 36834, `/tmp/v034_bench3.json`),
which ran concurrently with Claude Helper at ~32% CPU, produced the same
pattern: completion_tokens=57, decode_tps=0.437.

The contamination is structural, not transient. Waiting 10 minutes will
not clear it because the Claude app remains the active foreground process
throughout any executor invocation.

### Issue 2 — Model produces 57 tokens, not 64

With the "Once upon a time" prompt and `--max-new-tokens 64`, the model
consistently hits an EOS token at 57 generated tokens. This was also
documented in the v0.3.4 structural bench ("57 decode tokens"). The
`must be 64 tokens` requirement in the v0.3.5 prompt is therefore not
achievable with this prompt/model combination. Every trial would be
discarded under that rule, creating an infinite re-run loop.

This is a separate issue from contamination; it must be resolved
regardless of machine state.

## What attended work unblocks

Two fixes needed before re-issuing this prompt:

1. **Executor invocation from a quiet shell, not from Claude Code.**
   The bench must be run in a session where Claude Helper and Claude
   Renderer are NOT active. Options: (a) run the executor from a plain
   terminal session with the Claude desktop app closed, (b) invoke via
   a cron-style shell that runs when the user's GUI session is idle,
   (c) add an explicit "Claude processes < 10% CPU" check to the
   quiet-machine gate so the executor self-detects this class of
   contamination and halts before wasting a trial.

2. **Adjust the token-count requirement.**
   The v0.3.5 prompt requires `decode_tokens_emitted == 64` but the
   model consistently produces 57 tokens for this prompt. Either:
   (a) change the bench prompt to one that generates ≥ 64 tokens, or
   (b) relax the rule to `tokens_emitted == expected_completion` (where
   expected is verified empirically) and fail only on genuine
   truncations (decode_ms ≈ 600,000ms timeout). The v0.3.4 structural
   bench documented "57 decode tokens" without flagging it as a problem;
   the v0.3.5 prompt's 64-token rule should be reconciled with that.

## Followups

- Add "Claude Helper / Claude Renderer < 10% CPU" to the quiet-machine
  gate in future bench prompts — these processes are always active when
  the executor runs inside Claude Code and will contaminate any GPU bench.
- Verify the expected token count for the bench prompt so the
  "< max_new_tokens → discard" rule only fires on real truncations.
- Consider whether the bench should be scheduled as a cron-fired
  executor (running when the user's machine is idle, not in an active
  session) rather than triggered interactively.
