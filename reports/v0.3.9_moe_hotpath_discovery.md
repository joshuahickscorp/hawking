## Hot-path dispatch structure

`moe_block_batched_indexed_metal` (kernels/mod.rs:2427-2530) contains **ONE `ctx.dispatch_batch()`** call enclosing all of the following in sequence:

1. `encode_batched_gemv_indexed("moe_batched_gemm_q4_indexed")` — routed **gate** GEMV: shape routes×routed_mid×hidden, Q4_K weights, indexed per-expert in one batched call
2. `encode_batched_gemv_indexed("moe_batched_gemm_q4_indexed")` — routed **up** GEMV: same shape/kernel as gate, produces routed_up_out
3. `encode_silu_mul` — fused silu(gate) × up → routed_act: shape routes×routed_mid
4. `encode_batched_gemv_indexed("moe_batched_gemm_q8_0_indexed")` — routed **down** GEMV: shape routes×hidden×routed_mid, Q8_0 weights, projects activated experts back to hidden dim
5. [conditional: shared expert present] `encode_batched_gemv_indexed("moe_batched_gemm_q4_indexed")` — shared **gate** GEMV: 1×shared_mid×hidden, Q4_K
6. [conditional] `encode_batched_gemv_indexed("moe_batched_gemm_q4_indexed")` — shared **up** GEMV: 1×shared_mid×hidden, Q4_K
7. [conditional] `encode_silu_mul` — shared silu activation: shape shared_mid
8. [conditional] `encode_batched_gemv_indexed("moe_batched_gemm_q6_k_indexed")` — shared **down** GEMV: 1×hidden×shared_mid, Q6_K
9. `encode_route_accumulate` — weighted accumulate: routed_out (routes×hidden) × route_weights + shared_out → final_out (hidden)

**Key structure facts:**
- Exactly ONE CommandBatch per MoE block (gate + up + silu + down + shared + accumulate all in one CB)
- No per-expert iteration at the batch boundary — expert indexing is handled inside the Metal kernel via route_ids_buf
- `down_proj` (step 4) is inside the **same** CommandBatch as gate/up/silu (steps 1-3) — no separate CB for down
- Shared expert (steps 5-8) is also inside the same CB as routed experts — one CB for the entire MoE block

## Recommendation for v0.4.0

The real hot path is confirmed: `moe_block_batched_indexed_metal` fires via `moe_block_batched_dispatch`, all experts (routed Q4_K gate/up + Q8_0 down, shared Q4_K gate/up + Q6_K down) are processed in one CommandBatch. The next wedge should target either (a) fusing the routed `moe_batched_gemm_q4_indexed` gate+up GEMV into a single kernel call to halve indexed dispatch overhead, or (b) replacing `moe_batched_gemm_q8_0_indexed` with a Q4_K down_proj variant if the model can be requantized, since Q4_K GEMV is faster than Q8_0 on M-series. Additionally, the `gemv_f32_moe` count=1768 should be investigated — if these are dense-layer experts that skip the batched path, they represent untargeted CPU overhead. Clean bench data (via `tools/bench/clean_bench.sh`) is needed before any performance comparison can be trusted.
