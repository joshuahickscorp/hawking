## Phase 6 readiness assessment

**Where Phase 5 landed:**
- 5A (batched verify) infrastructure complete but only helps spec decode
- 5B.1 (single-TCB) confirmed working — zero mini-TCB overhead
- 5B.2/5C.1/5C.2 shipped as opt-in infrastructure

**Next highest-leverage work (ranked by expected gain):**
1. **Enable 5C.1 f16 KV cache by default** after wiring deepseek_v2.rs to route through
   f16 KV kernels. Per-layer KV bandwidth halved → attention path faster.
2. **Full-path 5C.2 f16 x_norm** — wire MoE block GEMVs to read f16 x_norm
   (need f16-input variants of `moe_batched_gemm_q4_indexed_v2t`, `gemv_f32_moe`,
   and dense FFN GEMVs). Per-layer FFN activation bandwidth halved.
3. **Phase 6: Persistent GPU command buffers** — eliminate even the single TCB
   rebuild per token. Pre-encode a "template" TCB for each layer, rebind position-
   and KV-slot args per token via KernelArgBuffer. Targets the ~60% remaining overhead.
4. **Phase 6: Draft model** — replace n-gram proposer with a small distilled draft
   (q4 1.5B) that produces α=0.6–0.8 on natural text. The Phase 5A batched verify
   infrastructure is ready; only the proposer quality limits α on natural text.
5. **Phase 6: Apple Neural Engine dispatch** — requires CoreML integration and is
   higher risk than persistent buffers; should follow Phase 6 persistent-buffer success.

**Is 60 dec_tps achievable?**
With Phase 5 at ~16-17 dec_tps, reaching 60 needs ~3.5× from Phase 6. That requires:
- +~2× from persistent command buffers (eliminates TCB rebuild overhead)
- +~1.5–2× from f16 KV + f16 activations (bandwidth halved on hot paths)
- +~1.2× from good draft model (α=0.6 on natural text = 1.6× effective throughput)
Combined: 16 × 2 × 1.7 × 1.6 ≈ 87 dec_tps (theoretical; real gains smaller due to
  memory contention). 60 dec_tps is plausible with all three landing cleanly.
