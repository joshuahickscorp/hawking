# v2.1 architecture sketch — Apple Neural Engine integration

This is the architectural plan for v2.1, post-launch. **Not dispatched yet.**
This document exists to make the plan inspectable before any implementation
work starts.

## Why ANE is the right next lever

Per the v1.1.0 architecture audit (commit `3f124f9`) and the post-Phase-4
TCB trace:

```
tcb_commit (forward pass compute)  3.22s   (69.4%)
gemv_f16 (LM head matmul)          1.41s   (30.3%)
moe_batched_gemm_q4_indexed        4.1ms   (0.1%)
```

The LM head is **30% of decode time** as a single discrete cost. Apple
Neural Engine has 16 cores designed for exactly this kind of f16 matmul.
**llama.cpp doesn't use the ANE.** MLX uses it but only through Core ML's
high-level abstractions. dismantle has the opportunity to do this
specifically and well as a differentiator.

If LM head ANE integration works:
- LM head time potentially 1.5-2× faster (Apple's optimized f16 matmul)
- GPU is FREE during LM head compute → can pipeline forward pass for
  next token's attention prep
- Net e2e: +15-25% on V2-Lite if both wins compound

That would take dismantle from ~17 dec_tps to ~21-22, a real perf
delta on the bandwidth-bound architecture.

## Why this is hard

**ANE is only accessible through Core ML.** No direct API. The path:

1. Convert weights to Core ML format at engine load (one-time cost)
2. Submit inputs through Core ML's `MLModel.predict(from:)` API
3. ANE may or may not run the matmul — Core ML's compiler decides
   based on shape, dtype, model structure, and even thermal state

The undocumented parts:
- Whether ANE actually runs vs falls back to GPU/CPU
- Per-call overhead (Core ML wraps every prediction call)
- Memory transfer cost (input/output tensors)
- How to keep the model "warm" across many short predictions

Apple provides essentially zero documentation for this use case.
There's reverse-engineering work (e.g., the `coremltools` Python
package) but no Rust ergonomic wrapper exists.

**Realistic probability of success:** 40-60% that ANE integration
lands cleanly. 25% that it lands but doesn't beat custom Metal kernels.
15% that we get it working with measurable wins.

This is genuinely speculative work, but the upside is unique to
dismantle's positioning.

## The technical approach

### Sub-phase 2.1A — Core ML FFI shim (~1-2 weeks)

New file `crates/dismantle-core/src/ane/coreml_ffi.rs`. Use existing
`objc2 v0.5` raw `msg_send!` to access Core ML's Objective-C API.

Required symbols (minimum viable):
- `MLModel` — load and predict
- `MLModelConfiguration` — set `computeUnits = .neuralEngine`
- `MLDictionaryFeatureProvider` — input wrapper
- `MLMultiArray` — tensor wrapper
- `MLFeatureValue` — value wrapper

Apple's headers are at `/System/Library/Frameworks/CoreML.framework/Headers/`.

**Smoke test in module:** create a 64×64 × 64 f16 matmul Core ML model
in code (no .mlmodel file needed), submit, verify output matches naive
matmul at atol=1e-3.

### Sub-phase 2.1B — Convert LM head to Core ML format at engine load (~1 week)

The hard part: dismantle stores LM head weights as f16 in mmap'd GGUF.
Core ML wants its own `.mlmodelc` compiled format.

Two approaches:

**Approach 1: Pre-compile .mlmodelc once, ship with weights**
- User runs `dismantle convert --weights model.gguf --out-mlmodelc model.mlmodelc`
- Bundles the LM head as a separate Core ML model
- At engine load, load both GGUF and .mlmodelc
- Pro: clean separation. Con: extra step, extra files to distribute.

**Approach 2: Build Core ML model in memory at engine load**
- Use Core ML's runtime model builder API
- One-time ~5-10 sec compile cost at first load
- Subsequent loads use the OS's compiled cache
- Pro: no extra files. Con: model-builder API is more complex.

Recommend Approach 2 for dismantle's single-binary positioning. Build
once at engine load (cached after first time).

### Sub-phase 2.1C — Wire ANE LM head into forward pass (~1 week)

In `forward_token_final_norm`:
- When `lm_head_schedule="ane"` is set in profile
- After final rmsnorm: feed x_norm into Core ML predict
- Get logits back as MLMultiArray
- Convert to f32 buffer for sampling

Profile flag: `lm_head_schedule="ane"` (default `"metal-argmax-token-only"`).

**Bench-first commit gate:** ANE LM head per-call must beat the existing
Metal `gemv_f16_metal_pinned` (3618 μs) by ≥1.5× in `bench-kernel`. If
yes, run e2e bench; flip default if e2e gain ≥ +5%.

### Sub-phase 2.1D — Extend to attention projections (~1-2 weeks)

If 2.1C lands cleanly, extend the same pattern to q_a/kv_a/q_b/o_proj
attention projections. These are smaller matmuls (1024-4096 wide vs LM
head's 102k-2048) but four of them per layer × 27 layers.

Per-shape Core ML models for each. Pipeline parallel: GPU runs MoE for
layer N while ANE runs attention projections for layer N+1's prep.

This is where the meaningful e2e gain comes from. 4 attn projections ×
27 layers = 108 ANE-runnable matmuls per token. Even small per-call
wins compound.

### Sub-phase 2.1E — Phase 6 close + bench

Authoritative bench. Update README with new perf numbers. Document
what worked + what didn't.

## Honest stack math

If everything lands at projected ratios:

| sub-phase | what | expected gain |
|---|---|---:|
| 2.1A | FFI shim (foundation) | 0% (infrastructure) |
| 2.1B | Build LM head Core ML model | 0% (still calling Metal at runtime) |
| 2.1C | Route LM head through ANE | +5-10% (LM head 30% slice × 1.3-1.5×) |
| 2.1D | Attn projections through ANE + pipeline | +5-15% (small per-call but compounded) |
| **stack** | | **+10-25% e2e** |

**Realistic v2.1 close: 19-21 dec_tps** (from current 17.3). Not 60.
But meaningful progress and a unique-to-dismantle architectural angle
worth talking about.

## Things that could fail

**Realistic failure modes, in priority order:**

1. **Core ML compiler doesn't run on ANE for our matmuls.** Apple's
   compiler is heuristic-driven. Small matmuls or specific shapes may
   silently fall back to GPU/CPU. Detection: profile with Instruments
   "Core ML" template; if no ANE rows show up, we're not actually using
   ANE.

   *Fallback:* land Path 2 (hand-rolled `simdgroup_matrix_multiply_8x8`
   Metal kernel) for the same matmuls. ~1.05× win without ANE.

2. **Per-call Core ML overhead exceeds savings.** Each Core ML predict
   call has setup cost (input wrapping, output unwrapping). If overhead
   is >3 ms per call, ANE LM head loses to Metal even if the matmul
   itself is faster.

   *Fallback:* keep Metal for LM head; only use ANE for batched
   workloads (multi-stream serving) where setup amortizes.

3. **Pipeline parallelism doesn't materialize.** GPU + ANE concurrency
   requires Apple's scheduler to actually run them in parallel. May
   serialize them depending on power state, thermal state, or backend
   choices we can't control.

   *Fallback:* document the unrealized parallelism; ship sequential
   ANE LM head as the actual gain.

4. **Memory pressure from dual model storage.** If we keep weights in
   both GGUF mmap AND Core ML format, we double the LM head's RAM
   footprint (~400 MB for V2-Lite vocab × hidden f16). Acceptable but
   notable.

   *Mitigation:* design Core ML model to point at the same memory if
   possible; otherwise accept the duplication.

## What ships if 2.1 fully STUCKs

Same as v2.0. The work isn't shipped if it doesn't deliver. We have
methodology infrastructure to gate this.

If 2.1A lands but 2.1B/C/D don't: ship the FFI shim as research
infrastructure. v2.2 can build on it.

If 2.1A doesn't land: write detailed BLOCKED report. Reframe v2.1
around AMX coprocessor (simpler, smaller wins, lower risk).

## Non-ANE alternatives if ANE proves infeasible

If 2.1A FFI shim takes >2 weeks of fighting with Core ML, pivot to
**v2.1 alternative track:**

### v2.1-alt-A — AMX coprocessor (~2 weeks, lower risk)
Apple Matrix coprocessor accessed via Accelerate framework. Used by
vDSP/BLAS. Small matmuls (gate router, sampling argmax) would benefit.

- Add `accelerate-sys` (Apple-shipped, no external dep)
- Use `vDSP_maxvi` for f32 argmax
- Use `vDSP_mmul` for gate router matmul
- Pipeline with GPU MoE compute

Expected: +3-5% e2e. Easier landing than ANE. Honest middle ground.

### v2.1-alt-B — true K-parallel batched verify (~3-4 weeks)
The other major lever. Currently `forward_tokens_batched_tcb` is
sequential-in-CB. Make it actually K-parallel by:
- Extending MLA decode to process K queries in parallel kernel launches
- Extending MoE GEMV to dispatch K tokens' active experts in parallel
- True K× compute amortization for spec decode

Expected: spec decode finally delivers wins on natural text. +20-40%
e2e for users with spec-friendly workloads.

### v2.1-alt-C — distilled draft model (~3-4 weeks)
Train a small (1B-2B) draft model that approximates V2-Lite. Use it
in the speculative decode pipeline. Gives spec α=0.4-0.6 on natural
text vs current α=0 (n-gram).

This requires model training infrastructure (PyTorch + dataset +
compute) outside of dismantle. Not in-session work for Sonnet/Opus.
Park as a parallel workstream — manager handles training, dismantle
provides the runtime hooks.

## Recommendation

**Start with v2.1A (Core ML FFI shim) after launch settles.** That's
the gating piece. If FFI works, continue 2.1B-D. If FFI fails after 2
weeks, pivot to v2.1-alt-A (AMX) which is lower-risk and still delivers
real gains.

**Don't start v2.1 until v2.0 has been launched and had ~1 week of
real-user feedback.** Per the v2.0 release checklist, the post-launch
retro should inform v2.1+ priorities. User signal beats theoretical
projection.
