# Phase 3 Acceptance Measurement Methodology

## Goal

Decide whether speculative decoding (Phase 6) is worth building. The gate is
**α ≥ 0.35**, where α = acceptance rate of the shared-expert-only draft model
against the full MoE model.

## Why shared-only as draft

DeepSeek-V2-Lite runs 2 shared experts + top-2 routed experts per MoE token.
The shared experts alone are ~30% of MoE compute; if they agree with the full
model ≥35% of the time, spec decoding yields net throughput gain
(`speedup ≈ α / (1 + draft_cost_fraction)`).

## Measurement design

### Draft model

`forward_token_shared_only`: identical to `forward_token` except in every MoE
layer the routed-expert contributions are zeroed. Gate logits and topk_gate
are still computed (routing decisions made but contributions skipped) so the
compute cost is realistic.

### Corpus

5 fixed prompts covering code, prose, factual recall, dialog, and technical text.
200 decode steps per prompt (or until EOS). Prompts are deterministic (see
`src/spec_acceptance_measure.rs`).

### Measurement protocol

At each decode step `t` with current sequence `s_0..s_t`:
1. Run `forward_token_shared_only(s_t, t)` → draft logits → `argmax` → `d_t`
2. Run `forward_token(s_t, t)` → full logits → `argmax` → `f_t`
3. `agreed += (d_t == f_t)`
4. Advance with `f_t` (full model drives the sequence)

`α = agreed / total_steps` across all prompts.

**Note on KV cache**: `forward_token_shared_only` at position `t` reads slots
`0..t-1` from the shared KV cache (same as full). Because KV writes differ
(attn output at shared-only vs full path), we run shared-only first and full
second so the KV cache reflects the full model state for the next step.

### Decision rule

| α | Decision |
|---|----------|
| ≥ 0.35 | Proceed to Phase 6 (speculative decoding) |
| < 0.35 | Skip Phase 6, log reason in `reports/phase3_decision.md` |

## Expected result

Based on literature (DeepSeek-V2 tech report) and analogies with Medusa-style
draft heads, α ≈ 0.5–0.7 for shared expert draft on familiar domains. If the
model's routing is highly variable, expect α closer to 0.35.

## Invocation

```bash
./target/release/dismantle-spec-acceptance-measure \
    --weights models/deepseek-v2-lite-q4.gguf \
    --steps 200 \
    --kernel-profile profiles/deepseek-v2-lite-q4.m3pro18.json
```

Output format:
```
prompt  0: steps=200  agreed=142  acceptance=0.7100  [fn fibonacci(n: u32) ...]
...
aggregate: steps=1000  agreed=620  acceptance=0.6200
phase3_acceptance=0.6200
```

The `phase3_acceptance=<val>` line is machine-parseable for CI scripts.
