# v2.1.0 T1.1 — kernel attribution report (post-T2.1 baseline)

**Generated:** 2026-05-13 23:32 PT
**Trace:** split-CB mode (`DISMANTLE_TCB_TRACE=gpu`), V2-Lite Q4_K_M,
4 tokens decoded under coexist conditions (slm trainer concurrent).
**Tool:** `tools/bench/analyze_tcb_trace.py`

This is the authoritative breakdown of where decode-time GPU work goes
post-T2.1, replacing the pre-T1.1 estimates from
`reports/v2.1.0_overnight_haul_closeout.md §"Updated bottleneck
attribution"` which were inferences rather than measurements.

## Method

T1.1 (`v2.1.0-T1.1` commit `85b7cb7`) added a `DISMANTLE_TCB_TRACE=gpu`
diagnostic mode that splits each TCB dispatch into its own
`MTLCommandBuffer` and reads `GPUStartTime/GPUEndTime` directly to
populate `DispatchSample::gpu_us`. T1.1b (commit `2fcd52e`) expanded
`static_kernel_name` to cover all 88 declared shader kernels, killing
the previous trace's 36% "other" bucket.

## Caveat (read before drawing conclusions)

**Split-CB mode breaks GPU pipelining.** Each dispatch pays a full
commit/sync. Kernels that normally overlap with predecessors get
over-attributed; the absolute %s are skewed vs the production
single-CB-per-block reality. **Relative ordering is reliable** for
picking the next kernel target; **absolute % attributions are NOT** —
production share for any given kernel will likely be lower.

This trace was also taken under coexist conditions (slm trainer
concurrent), which contaminates absolute timing further. The per-kernel
GPU times remain comparable to each other within the trace; they
should not be compared across traces taken in different conditions.

**Biggest correction:** LM head is **3%**, not 7-9%. The T2.8 attempt at
simdmat LM head failed because the LM head is already small. The
v1.1.0 plan's "attack the LM head, 30% of decode" mis-attribution
continued to bias estimates upward even after the fix; the actual LM
head slice is too small to matter for top-line dec_tps until MoE is
much faster.

## Next Tier 2 candidates (ranked by hypothesized e2e gain)

### T2.11 — port v2t to Q5_0 routed-down kernel (TOP CANDIDATE)

**Why:** `moe_batched_gemm_q5_0_indexed` is 16.28% of GPU time in this
trace and runs the **un-optimized basic v1 kernel**: 1 row per TG, no
threadgroup x-cache, tree-reduction in shmem, per-element decode in
inner loop. The Q4_K equivalent (`moe_batched_gemm_q4_indexed_v2t_gu_v2`)
uses 8 rows per TG, threadgroup x-cache, simdsum, and paired nibble
reads — the techniques T2.1 landed.

**What:** add `moe_batched_gemm_q5_0_indexed_v2t` mirroring the Q8_0
v2t kernel's structure (`shaders/moe.metal:872–919`) with the Q5_0
dequant inner loop (`q5_0_value` semantics inlined). Plumbing parallel
to `moe_batched_gemm_q8_0_indexed_v2t`: wire schedule mapping in
`routed_down_kernel`/`shared_down_kernel`, add to `static_kernel_name`,
add a parity test.

**Hypothesized gain:** Q5_0 currently uses ~38ms/token in split-CB.
v2t pattern delivers ~2-3× on Q4_K (per T2.1's win). If Q5_0 follows
the same speedup curve, ~+8% e2e on V2-Lite, scaling smaller in
production single-CB-per-block where MoE share is lower.

**Risk:** medium. Q5_0's 5-bit packing crosses byte boundaries (high
bits are in a separate qhi field), making the per-simd-lane decode
slightly more complex than Q4_K nibble extraction. Parity test against
the existing kernel is critical.

**Estimate:** 2-4 hours implementation + parity test + bench.

### T2.12 — port v2t to Q6_K shared-down kernel

**Why:** `moe_batched_gemm_q6_k_indexed` is 5.54% of GPU time, also
un-optimized. Smaller share than Q5_0 but the shared expert runs every
token regardless of routing, so it's always on the hot path.

**Hypothesized gain:** +2-4% e2e.

**Risk:** higher. Q6_K is a more complex super-block format (256
elements/superblock with scale+min fields) which is structurally
different from the per-block layouts the existing v2t pattern handles.
May require a different optimization strategy.

**Estimate:** 1-2 days.

### T2.13 — attn QKV fusion (T2.4 from the comprehensive plan)

**Why:** `rmsnorm_gemv_f32_attn_pinned` (9%) and `gemv_f16_simdmat` (9%)
both read the same activation vector for different attention
projections. Fusing the q_a + kv_a + q_b + o_proj dispatches into
one kernel per pair (or all four) eliminates the duplicated activation
reads. The closeout's T2.5 (rmsnorm pre-fusion) failed because GPU
work was already parallel; QKV fusion is different — it saves
*activation reads*, not redundant compute.

**Hypothesized gain:** +3-5% e2e.

**Risk:** medium. T2.5's lesson about parallel-is-free applies if the
fused kernel uses more threadgroup memory than the existing
rmsnorm_gemv_f32_attn_pinned; need to validate per-call bench gain
before flipping default.

**Estimate:** 1 week.

### T2.14 — investigate `gemv_f16` 7ms/call

**Why:** LM head dispatch is 1 call/token × 7000us = 7ms. That's
high for a single GEMV. The kernel reads 419 MB of fp16 weights
(2048 × 102400 × 2 bytes); at 130 GB/s peak that's ~3.2ms minimum.
Actual 7ms = ~55% bandwidth utilization, which is fine standalone but
in split-CB mode it has no contention, so the production number is
likely similar.

**Action:** check whether wiring a top-K cutoff (Phase 5B.2 kernel from
commit `6b4c95a`, removed in cleanup `098a5e7`) would actually help
greedy decode. Closeout already concluded it doesn't (greedy needs
full LM head). Confirmed by T1.1 numbers (3% of decode; even
eliminating it entirely is 3% e2e gain at most).

**Hypothesized gain:** ≤+3% e2e. **Deprioritized.**

## What T1.1 does NOT tell us yet

- **Per-shape bandwidth utilization.** Need per-kernel byte footprints
  (weight reads + activation reads) to compute effective bandwidth
  per kernel. Currently the analyzer computes a single aggregate
  number against the 1.82 GB/token figure.
- **Production single-CB-per-block attribution.** Split-CB mode skews
  toward kernels that wouldn't normally overlap. The morning needs to
  either bench production mode AS WELL or compare relative ordering
  across both modes to validate.
- **llama.cpp comparison.** T1.3 from the v2.1.0 plan was deferred.
  Doing it would tell us whether our Q4_K_v2t_gu_v2 is already at
  llama.cpp's level (in which case Q5_0 is the only port left) or
  there's still kernel-side headroom on the dominant Q4_K path.
