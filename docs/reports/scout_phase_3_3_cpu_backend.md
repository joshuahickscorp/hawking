## Decisions (most already implied by the existing path)
1. **GEMV over Q4_K: pre-dequant per call** (status quo, `matmul_q4_dispatch`). Keep it for v1 (correctness/reach). Dequant-on-the-fly (fused) is a *perf* refinement, not needed for the reach bar — do NOT scope it now.
2. **No new dep.** `gemm`/`matrixmultiply` would need user approval (CLAUDE.md: "Cargo.toml dependency additions need the user's approval") — **FLAG, do not pull.** v1 ships on the existing scalar `gemv_f32`. `std::simd` is **portable_simd = nightly-only** (unstable `#![feature(portable_simd)]`) → would force a nightly toolchain on the whole crate. **Recommendation: v1 = scalar (stable, builds everywhere). Defer SIMD.** If pursued, gate any `std::simd` behind `#[cfg(feature=...)]` + nightly, never on the default stable build path.
3. **CommandRecorder shape for CPU: there is none / immediate-exec.** The CPU path does **not** go through `forward_token_greedy_tcb` (`:3427`, macOS-only TCB) — it goes through `forward_token` which calls the `*_dispatch` helpers that execute **immediately** (no command buffer). This matches the 3.1 note "trait does NOT impose one-dispatch-one-commit"; CPU's `Recorder` is a no-op/eager executor. **Do NOT build a fake command buffer.**

## Ordered, measurement-gated attempts (cheapest/safest first)
1. **Provision a non-macOS toolchain + `cargo check`** (the oracle this scout couldn't run). Fix whatever the compiler flags. *Expected* blockers (static analysis — verify): (a) any `bytemuck`/`f16` cast paths; (b) `model/mod.rs:117` non-macOS create arm — confirm it returns a built engine, not an error stub, for qwen2; (c) the `metal_dispatch`-gated **MoE** path — deepseek/mixtral off-macOS `route_experts` returns `Ok(None)` (`deepseek_v2.rs:2209`), so **scope the cross-check to dense qwen0.5b only** (MoE CPU decode may be incomplete; out of 3.3 scope, log it).
2. **Add the force-CPU knob** (the one genuinely missing piece). Today `metal_ctx = MetalContext::new_with_trace(...).ok()` is **unconditionally `Some` on macOS** (`qwen_dense.rs:799`) → the CPU path is unreachable on a Mac, so the parity test can't run both arms. Add `EngineConfig.force_cpu: bool` (or env `DISMANTLE_FORCE_CPU=1`) that forces `metal_ctx=None` in every model's `load`. Smallest blast radius; needed by step 3.
3. **The cross-check test** (the deliverable): macOS-gated `tests/cpu_backend_parity.rs` (clone `integration_greedy_64.rs` shape). Load qwen0.5b **twice**: arm A normal (Metal `forward_token_greedy_tcb`), arm B `force_cpu` (CPU `forward_token`). Compare **per-token logits at atol=1e-3** (the plan's bar) for the first N tokens, AND first-3 greedy token IDs identical. (CPU `forward_token` returns `Vec<f32>` logits directly → easy diff; Metal arm via `forward_tokens_for_test` `:2740` which on macOS routes the TCB path. Confirm `forward_tokens_for_test` uses the GPU path on macOS — if it calls CPU `forward_token` on both, instead diff CPU-`forward_token` vs a GPU-logit capture.)
4. **Lift the hard gate** (`Cargo.toml:27`): keep `metal*` deps macOS-only (they already are — leave :27 as-is). The lift is in **code cfg**, not Cargo: the crate *already* compiles its CPU primitives unconditionally. Phase 3.3's "lift the gate" = make the **CLI/serve binary** build+run off-macOS (the `metal_dispatch` mod stays `#[cfg(target_os="macos")]` `kernels/mod.rs:367`; that's correct — it's never called off-macOS). Verify `crates/dismantle/src/main.rs` (subcommand stubs already paired, e.g. :813) and dismantle-serve have no remaining hard macOS-only call on the decode path.
5. **(Optional, perf, NOT the bar) std::simd `gemv_f32`/dequant** — only after 1-4 are green and only if the user approves nightly. Out of the reach gate.

## EXACT parity gate + ship gate
- **Parity gate (this is the plan's 3.3 gate, §3.3):** CPU `forward_token` vs Metal decode on **qwen0.5b** (`models/qwen2.5-0.5b-instruct-q4_k_m.gguf`, 491 MB, present): per-token logit **atol=1e-3** (fp16 floor per CLAUDE.md) **+ first-3 greedy token IDs identical**. NOT bit-identity (CPU f32 dequant ≠ Metal predec fused FMA order → expect ~1e-3 drift; that's the floor, real bugs are orders larger). Plus the existing always-on golden `integration_greedy_64` must stay green on macOS (no Metal-path change).
- **Ship gate (§3.3):** builds on a non-macOS target (step 1 oracle) **AND** CPU decode produces correct output (parity green) end-to-end. **Perf is explicitly NOT the bar — reach is.** Do not paired-bench; the CPU path is ~100× slower and that's fine.

## Kill cross-check (dead_levers.md)
- **Adjacent but DISTINCT — not killed:** `dead_levers.md` "Non-GEMM CPU P-cluster offload (#5, Type-1)" and "Hybrid AMX+GPU GEMM (#1, Type-1)" kill **CPU/AMX as a *decode-speed lever on the M3 Pro hot path*** (decode non-GEMM = 3.2% of wall, no dependency-free slot; AMX can't eat Q4_K). **Phase 3.3 is NOT that lever.** It is *portability/reach* — a CPU backend so the engine **runs off-macOS at all**, where there is no GPU. The kills are about *adding* CPU work *alongside* Metal *to go faster*; 3.3 *replaces* Metal *to exist elsewhere, accepting slower*. Quote the #5 resurrection: *"a fully-parallel CPU-side speculative draft … a different problem"* — confirms cross-engine CPU-for-speed is dead, says nothing about CPU-for-reach. **Distinct → live.**
- **No Type-1 kill covers off-macOS reach.** The whole silicon audit is M3-Pro-shared-bus-specific. No entry forbids a CPU `impl Backend`. ✅ genuinely-live.
- The std::simd sub-form: no kill exists, but it's a *perf* refinement on a *reach* task — defer regardless.

## ⚠️ STRATEGIC CAUTION (orchestrator)
- **This is a REACH/PORTABILITY win — zero tps, zero J/tok, zero long-context.** It does not move either north-star metric. Its value is the llama.cpp-parity gap in Part III ("dismantle runs on exactly one platform") + it is the **CPU-fallback substrate** the `ggml_backend_sched` lesson (Part V.1) says makes partial GPU backends shippable (Phase 4.4).

Second risk: MoE (deepseek/mixtral) CPU decode looks **incomplete** off-macOS (`route_experts → Ok(None)`) — **scope 3.3 to dense qwen0.5b**, log MoE-CPU as a separate follow-up. Third: `forward_tokens_for_test` may call the *same* CPU `forward_token` on both arms (no Metal contrast) — verify before building the test, else capture GPU logits explicitly.
- **Smallest real new code:** `force_cpu` knob (~5 models × 1 line) + one cross-check test. The "backend" is already on disk.
