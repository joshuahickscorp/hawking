# Phase 2.3 scout — flash-style decode attention (GQA dense) (scout, 2026-06-01)

> READ-ONLY scope. Replace the materialize-all-scores `mha_decode_f32` (`mha.metal:34`, 2.92% short-ctx GPU per 0.2) with an online-softmax flash-decode kernel for the **Qwen dense GQA** path. Verified against the tree at `paradigm/exec`.

## ★ The decisive find: the kernel already exists for MLA — port it, don't invent it
A full Flash-Attention-v2 online-softmax decode kernel is **already in-tree and shipped on the deepseek MLA path**: `flash_attn_decode_kernel` (`attn.metal:536-681`), wrapper `flash_attn_decode_metal` (`kernels/mod.rs:3781`), wired at `deepseek_v2.rs:3574`, pipeline-registered `metal/mod.rs:465`, parity-tested `tests/v1l_flash_attn_parity.rs` (atol 1e-3 vs `mla_decode_metal`). It implements exactly the spec'd algorithm: running max `state[0]`, running sum `state[1]`, correction `acc *= exp(m_old−m_new)`, tiled by `FLASH_TG=128`, no `scores[seq_len]` buffer. **This is your template.** The only reason it doesn't cover Qwen: it reads the MLA *latent* KV (`c_kv`/`k_pe` + on-the-fly `w_uk`/`w_uv` projection), not the raw `(seq, n_kv_heads, head_dim)` K/V the dense path stores. So the task is a **GQA re-skin of an existing, validated kernel**, not a green-field flash kernel.

## What the dense kernel does today (the thing to replace)
`mha_decode_f32` (`mha.metal:34-113`): one TG/head (TG=128, `kernels/mod.rs:5818`), 4 serial phases — materializes `scores[t]=dot(q_h,K[t,kv_h])·scale` into shmem (`:59-66`), tree-max (`:69-82`), exp+sum in place (`:85-100`), V-weighted sum (`:103-112`). Shmem = `(seq_len+TG)·4 B` (`kernels/mod.rs:5819`). **Hard wall: at the 32 KB TG-mem ceiling this caps `seq_len ≈ 7800` tokens** (kernel comment `mha.metal:23` says ~8000). Flash removes the O(seq) shmem → O(1) `state[8]`+`scores_tile[128]` → no context cap and more concurrent TGs/core at long ctx.

Run `integration_greedy_64` golden — but **expect it to differ** from the materialize-path golden by sub-atol drift; if the 64-tok hash moves, that is the *reduction-reorder*, not a bug — do **not** auto-rebaseline; record the divergence and gate on the 3-token + `atol/rtol` numerical test instead (flash is explicitly not bit-identical, so a golden-hash gate is the wrong tool here).

## Long-context paired bench setup (name it)
- **Harness:** `tools/bench/long_context_bench.sh` (exists; decodes at `CTXS="4096 16384 32768"`, `GEN_TOKENS=32`, synth code prompt, locked profile). It is **absolute** (single-arm) — for the **paired** gate, run it twice (flash-on env vs flash-off env) and diff per-ctx `decode_tps`, OR extend `paired_lever.sh --env-a/--env-b` to take a long synth prompt. Report range + 95% CI + IQR per ctx; tag measured (Claude open → ratios only, never absolute tps).
- **Where the crossover is (model it, then measure):** per-token weight read ≈ 1.93 GB (fixed). Per-token KV read (f32, all 36 L, 2 kv-heads, head_dim 128, K+V) = `2·36·2·128·4·seq` B = **147 KB·seq**. KV-read rivals weights at seq ≈ `1.93e9 / 147e3 ≈ 13.1K tokens` (f32 KV). Flash doesn't cut *bytes* (it still streams all K/V once) — it cuts **shmem footprint → occupancy** and removes the **~7800-token shmem cap**. So the realistic flash win is **(a) anything past ~8K is impossible today → flash is an enabler, not a speedup, there**, and **(b) a modest occupancy tps bump in the ~2K–8K band where attention's GPU share has grown from 2.9% but shmem still fits.** Bench `CTXS="2048 4096 8192 16384"` and report the share-vs-ctx curve.

## Risks / strategic caution (blunt)
- **This is a long-context + energy + reach lever with ~0 short-context tps headline.** 0.2 is unambiguous: attention is 2.92% of short-ctx decode; the ±3% position-bias floor means a short-ctx tps win is **structurally unmeasurable**, and the dead MLA-rewrite proves even a perfect short-ctx attention rewrite is sub-gate. Do **not** sell this as closing the 1.55× llama gap — that gap is 86.7% the predec GEMVs (scout 2.1), untouched by this.
- **Realistic ceiling:** at short/mid ctx, ~0 to a few % (occupancy). The *real* value is **removing the ~7800-token hard cap** (`mha.metal:23`) → dismantle simply **cannot decode past ~8K today**; flash makes long-context decode *possible at all*. That's a **capability/reach unlock**, not a tps number.
- **Biggest risk: there is no long-context workload that matters yet.** Per CLAUDE.md scope-discipline and the kill's resurrection clause, the honest call is **build it when a long-context workload is on the table, not now.** The 0.2 Phase-2 order ranks this **last (4th)** for exactly this reason. Cheap insurance: the kernel is a ~1-hour port of an existing validated template, so the *spike* (attempt 1: kernel + parity test only, no wiring) is low-cost and de-risks the eventual need — but **wiring/shipping should wait** unless the user has a >8K-context use case. If you must justify building now: the only defensible reason is "we want to advertise >8K context support"; absent that, **HOLD after the parity-green spike** and record the kernel as ready-but-dormant behind `DISMANTLE_QWEN_FLASH_ATTN`.
- **Profile shader-hash gate (will bite, same as 0-setup `64edfbd`):** any edit to `mha.metal`/`attn.metal` changes `shader_source_hash()` (`profile.rs:392`) → the guard at `profile.rs:223` rejects `profiles/qwen3b-instruct-q4k.m3pro18.json` ("shader hash mismatch") and **every `--kernel-profile` bench writes no JSON**. Re-stamp the single `shader_hash` field (the `d168542`/`64edfbd` chore), do **not** re-run autotune (contaminated timings). Required before any bench in attempts 2-3.
