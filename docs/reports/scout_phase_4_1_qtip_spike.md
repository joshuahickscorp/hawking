# Phase 4.1 spec — QTIP-on-Metal decode-cost spike (scout, 2026-06-01)

> The DECODE half of the Phase 4.1 gate (plan:193-196). Quality is the separate `colab/05_combined_quality_gates.ipynb` oracle (leaning NO-GO, see kill cross-check). This spike answers ONE question: does a 3INST/1MAD trellis decode GEMV stay **BW-bound** on M3 Pro, or land compute-bound like the dead Q3_K kernel? `[serial][high-risk]`.

## Prior (state it up front, per charge)
**Strong prior: NO-GO (expect a Type-1 kill).** Three converging signals: (1) Q3_K is **Type-1 compute-bound at 22-24% peak** (`dead_levers.md:178-185`) — the wall is per-element decode ALU, not bytes; (2) a trellis is *more* per-element ALU than f16-predec-Q3, which is itself the *leaner* reframe of the dead Q3 (`qtip_bytecut_design:166-178, 4.2`); (3) the trellis adds a **serial state dependence** (`state[i]←state[i-1]`) with no Q4_K analog — Q4_K nibbles are independent (`qtip_bytecut_design:286-294`). The decode half is here precisely to kill cheaply *before* spending Colab compute on the codec.

## Layout / reference facts (verified)
- **Dead reference (compute-bound):** `gemm_q3_k_fused_v2` inner loop `quant.metal:366-376` — calls `q3_k_value` (`:39-57`) → `q3_k_scale` (`:26-37`, branch + nibble-select + shift/mask + OR + −32 per element). 110 B/block, **33-36 GB/s = 22-24% peak**.
- **BW-bound reference (target):** `gemm_q4_k_v4_predec_pair` `quant.metal:2219-2280` (45.4% of decode, ~52% peak) and `gemm_q4_k_v4_predec_2r` `:2358` (41.3%, ~56% peak). Inner FMA `:2259-2267`: contiguous 32 B/simdgroup nibble load, predecoded scales, **independent** nibbles. This is the outer-GEMV structure QTIP reuses; only the inner decode changes.
- **Block bytes target:** QTIP-3.0 ≈ 96-104 B/block, **no scale plane** (self-describing) vs Q4_K-predec-f16s 176 / f32 208 (`qtip_bytecut_design:212-218`).
- **Harness:** `tests/q3k_bytecut_bench.rs` — `time_dispatch` (`:108`, WARMUP=30/ITERS=200, fresh TCB commit_and_wait = one decode round-trip), `gbps` (`:131`), `bench_shape` (`:135`), 3 Qwen shapes (`:264-266`: 2048×2048, 11008×2048, 2048×11008). `make_x` (`:97`) reusable; needs a new `make_qtip_bytes`.
- **Wrapper template:** `gemv_q3_k_v4_predec_pinned_tcb` `kernels/mod.rs:1457-1520` (TG=256, ROWS_PER_TG=8, `expected_bytes` guard, buffer binds). QTIP wrapper drops the `scales_buf` arg (self-describing) and uses block stride 96-104.
- **Kernel registration:** match arm `metal/mod.rs:442-453` (alongside the Q3/Q4 predec names). Flag `DISMANTLE_QWEN_QTIP`, `OnceLock`, default-off — but this spike is microbench-only, no dense wiring.

## EXACT gates (per plan + design §5.2)
- **Parity gate (4.1-1):** this is a **microbench, not a correctness gate** — it measures wall time / bytes only, ignores numerical output (same as `q3k_bytecut_bench.rs:26-30`). The trellis decode has no bit-identity claim (different codec); a *separate* `tests/qtip_trellis_parity.rs` vs a CPU-ref QTIP decode at **`atol=1e-3` fp16** is required only if 4.1 clears and 4.2/4.3 proceed (`qtip_bytecut_design:6.5`). Do NOT gate the spike on parity.
- **Ship/proceed gate (the decode GATE):** trellis kernel runs **≥~55-60% of peak BW** (BW-bound) AND beats Q4_K-predec µs/call on the dominant `_pair`/`_2r` shapes → **proceed to 4.2/4.3** (plan:196 says >~60%). Clean-room absolute only (Claude open inflates; §A uses 45% GO / 35% NO-GO on the f32-predec-Q3 proxy line).
- **Kill gate:** lands **~24-30% peak** (à la Q3_K) → **STOP, skip 4.2-4.3, record in `dead_levers.md`.** Classification: **Type-1** if the simdgroup model can't hide the trellis ALU even with lane-independent sub-blocks (the expected outcome — the serial-dependence + ALU-density is then a measured GPU-model fact); **Type-2** only if a *specific* different layout (e.g. wider lane-independent state, a fused predec-of-trellis-seeds) is named that might flip it, with its oracle.

## Kill cross-check (mandatory)
- **`Q3_K sub-Q4 decode byte-cut` — Type-1 kill (`dead_levers.md:178-185`):** clean-room 22-24% peak, "compute/residual-bound on the inline scale + hmask decode, NOT bandwidth-bound." **QTIP is its NAMED Type-2 reframe** (`:183`: "gather-free trellis, arithmetic-coded, no hmask residual, no LUT"). **Is my scoped form distinct?** Mechanically yes — no hmask, no 6-bit inline scale, no scale plane. But it sits on the **wrong side of the same transform**: it *re-adds* per-element compute (`qtip_bytecut_design:174`) that predec *removed*. So the spike is legitimately live *as a measurement* (the reframe is named-alive in the ledger), but the prior is that it dies for the **same reason** Q3_K did. The resurrection check (`:185`) routes the byte-cut axis through QTIP only — this spike IS that route's decode gate.
- **`Learned per-model codebook` Type-1 (`:116-120`) + `Low-rank codec` Type-1 (`:107-112`):** dead on Apple's no-hardware-gather wall. QTIP survives these **by construction** (codes computed, not gathered) — confirmed `:120` resurrection check names "LOOKUP-FREE bitshift trellis" as the lone survivor. My kernel is gather-free (contiguous bitstream, coalesced like the nibble plane) → **distinct from these, correctly.** Do not re-scope a gather codebook.
- **`Decode-kernel micro-opt A5/A6` + `A10 layout` Type-1 (`:208-226`):** the Q4_K predec GEMV is at the Apple memory-model optimum; remaining dense-tps headroom is "fewer bytes (QTIP) or spec/stateful, NOT decode-kernel micro-opt." QTIP is the *fewer-bytes* lever, not a micro-opt of the existing kernel → distinct.
- **Quality leg (`dead_levers.md:361-366`):** `bits_needed=[+1.37,+0.44]` both positive ⇒ ~0.44 bits short of Q4_K_M even best-case; decisive Cornell codec NOT run (`ALLOW_FRESH_QTIP_CODEC=False`). **This spike does NOT touch quality** — even a BW-bound decode is moot if the codec can't hit Q4_K_M quality on code. Flag to orchestrator: 4.1-decode GO does **not** unlock 4.3 alone; the Colab quality oracle must also clear.

## Strategic caution (blunt)
- **Realistic ceiling:** if it clears (low-prior), −41 to −45% bytes on the 86.7%-of-decode GEMVs ≈ the ~1.55× llama gap on paper — a **both-metrics win (tps + energy)**, the single biggest byte lever on the board. This is the *only* surviving sub-Q4 byte-cut bet.
- **Biggest risk:** the serial trellis state-dependence (no Q4_K analog) — if the Colab encoder can't emit lane-independent sub-blocks, the decode is **structurally worse than Q3** on this GPU, not better. That's a co-design risk you can't fully retire in a microbench with synthetic blocks (you control the synthetic layout, so a passing microbench is necessary-not-sufficient; the real encoder must reproduce that layout).
- **What kind of win:** primarily **tps + energy** (fewer bytes on the dominant GEMV; energy tracks GPU-time per 0.3). Secondary **footprint** (~1.2 GB vs 1.9 GB on disk). NOT a long-context or portability win.
- **Honest recommendation:** take 4.1-0 (the free §A proxy, already NO-GO at 24%) as the cheap kill unless the orchestrator explicitly wants a direct trellis number. Building 4.1-1 is real kernel effort (novel hot loop, the parallelization problem) to most-likely confirm a Type-1 the proxy already implies. If built and it dies, record Type-1 and the byte-cut axis is **closed** (every formulation — Q3_K direct, low-rank, codebook, QTIP-decode — then dead); dense-tps headroom routes to dispatch-fusion (2.2) / spec / stateful, not bytes.
