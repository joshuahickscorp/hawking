# v2.1.0 T2.13 close — q_proj pinned as f16 (+6.6% e2e)

**Date:** 2026-05-14
**Branch:** `claude/modest-tharp-77cc98`
**Predecessor:** `60de3e9` (T2.12)

## What landed

A 6-line infrastructure change: pin V2-Lite's `q_proj` weight matrix
as f16 (instead of f32) at model load, and route the two TCB
attention call sites through the existing
`rmsnorm_gemv_f16w_attn_pinned_tcb` kernel instead of the f32 variant.

V2-Lite has `q_lora_rank=0` so the model uses the non-LoRA q_proj
path on every layer — making `rmsnorm_gemv_f32_attn_pinned` an
11.38% GPU hotspot in the post-T2.12 trace. Switching to the
already-existing f16-weight variant halves the q_proj weight read
bandwidth (25 MB → 12.5 MB per call) and converts the call into a
sibling of the kv_a_proj dispatch (which has been f16w-pinned since
v0.8.x).

## Why this works

`rmsnorm_gemv_f32_attn_pinned` and `rmsnorm_gemv_f16w_attn_pinned`
have identical Metal binding layouts and identical compute logic
(see `crates/dismantle-core/shaders/attn.metal:483` and `:537`). The
only difference is `device const float* w` → `device const half* w`,
so the weight buffer is read as 2-byte half values that are widened
to f32 inside the kernel.

For V2-Lite's q_proj at `[3072, 2048]`:
- f32 weight: 24 MiB
- f16 weight: 12 MiB
- per-call us in trace: 397 (f32) vs 94 (f16w) — 4.2× faster.

q_a_proj and kv_a_proj were already pinned as f16 via the same
helper (`crates/dismantle-core/src/model/deepseek_v2.rs:734`'s
`upload_f16`). q_proj was the only non-LoRA attention projection
still pinned as f32 — purely historical, no design rationale.

## Parity behavior

f16-pinned q_proj introduces one fp32→fp16→fp32 round trip per
weight (~1e-4 relative error per output). q_a_proj and kv_a_proj
have used this representation since v0.8.x without regression.

Smoke check at 32 tokens, "The capital of France is", greedy temp=0:
- baseline (T2.12): `Paris.\n\nThe capital of France is Paris.\n\nThe`
- T2.13:           `Paris.\n\nThe capital of France is Paris.\n\nThe`
- **bit-identical**.

## Files changed

- `crates/dismantle-core/src/model/deepseek_v2.rs`:
  - line 749: `upload(...)` → `upload_f16(...)` for `q_proj`
  - line 3435 (`attention_tcb_inner`):
    `rmsnorm_gemv_f32_attn_pinned_tcb` → `rmsnorm_gemv_f16w_attn_pinned_tcb`
  - line 3620 (`p1_into_tcb`): same swap.

No shader source changes; no shader_hash bump. No profile changes.

Legacy CPU pair-dispatch path (`gemv_f32_attn_pair_dispatch` at
line 1725) already guards against undersized pinned buffers
(line 1739 `buf_a.length() >= f32_a`) and falls back to the CPU
GEMV when the pinned buffer is f16-sized. This was already in place
for the kv_a_proj f16w transition; the same guard handles q_proj.
