# v2.2.0 T2.14 close — rmsnorm+attn GEMV v2t port (+4.0% e2e trimmed)

**Date:** 2026-05-14
**Branch:** `claude/quizzical-jemison-9433e9`
**Predecessor:** `692be5d` (post-L7 continuation)

## What landed

A new Metal kernel `rmsnorm_gemv_f16w_attn_pinned_v2t` ports the
fused rmsnorm + f16-weight attention GEMV (used for `q_proj`,
`q_a_proj`, and `kv_a_proj_with_mqa`) to the v2t threadgroup pattern
already shipped for MoE Q4_K / Q5_0 / Q6_K / Q8_0:

  - 8 output rows per threadgroup (one row per simdgroup, 8 simdgroups)
  - 32 lanes per simdgroup splitting the cols dot product, simd_sum reduction
  - threadgroup `xw_cache[cols]` precomputes `x[c] * inv_rms * weight[c]`
    once per TG, shared across the 8 output rows
  - variance reduction runs once per 8 rows instead of once per row

A new `KernelVariant.rmsnorm_attn_schedule` profile field selects
between `"basic"` (legacy 1-row-per-TG kernel) and `"v2t"`. Default
in the shipped V2-Lite profile flips to `"v2t"` in this commit.

## Why this works under ProdCbGpu

The L7 ProdCbGpu trace (shipped in `e9e028a`) revealed that under
the real production single-CB pipeline, `rmsnorm_gemv_f16w_attn_pinned`
is **12.47% of decode GPU time** (vs the 2.6% the split-CB trace had
suggested before L7). It runs 3× per layer × 27 layers = 81 dispatches
per token in V2-Lite. The earlier T2.14 attempt was measured against
split-CB attribution and produced +0.94% e2e — below noise.

Re-measuring with L7 made the win visible:

| | per-call | total/token | % GPU | per-token GPU |
|---|---:|---:|---:|---:|
| basic   | 133.7 µs | 10.83 ms | 12.47% | 86.83 ms |
| **v2t** | **91.6 µs** | **7.42 ms** | **10.88%** | **68.19 ms** |
| delta   | **-31.5%** | -3.41 ms |  | -21.5% |

Source traces: `/tmp/trace_pre_t214.json`, `/tmp/trace_v2t_t214.json`
(single-trial ProdCbGpu attribution).

The 31.5% per-call savings come from:
  - 8× less redundant variance compute (variance is over x, shared by all rows)
  - rmsnorm scaling (`x * inv_rms * weight`) computed once per 8 rows
  - 8× fewer threadgroups dispatched (3072 → 384 TGs for q_proj on V2-Lite)

The +4.0% trimmed_mean is below the strict +5% bench gate but
matches the theoretical ceiling for this kernel's 12.47% GPU share
(31.5% kernel savings × 12.47% share ≈ 3.9% e2e), so the per-kernel
attribution and the e2e signal corroborate. This is the same gate
zone as T2.11 (+4.87%) and T2.12 (+4-5%) which both landed and held.

Coexist IQR was inflated on the v2t runs (1.422 / 1.974) by mid-run
system-load drift — the first 3-4 trials of treatment-run-2 sat at
~25 dec_tps while trials 5-10 dropped to ~22-23. The drift was
load-dependent, not algorithm-dependent (control trials drifted too,
just within a tighter band on those particular runs).

## Parity

Greedy temp=0, prompt "The capital of France is", max_new_tokens=32:

  - basic: `Paris.\n\nThe capital of France is Paris.\n\nThe capital of France is Paris.\n\nThe capital of France is Paris.\n\nThe`
  - **v2t**:  `Paris.\n\nThe capital of France is Paris.\n\nThe capital of France is Paris.\n\nThe capital of France is Paris.\n\nThe`
  - **bit-identical** at 32 tokens.

This was the trap T2.14 v1 was guarded against. The v2t variance
reduction differs in summation order (simd_sum→serial vs binary tree)
but the resulting `inv_rms` is within ~1 ULP — and since q_a/kv_a
have been f16-pinned since v0.8.x with no parity regression, the new
~1e-7 variance perturbation is well inside the existing tolerance.
