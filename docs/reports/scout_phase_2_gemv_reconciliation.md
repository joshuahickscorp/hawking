# Scout — Phase 2 GEMV adversarial reconciliation (scout, 2026-06-01, branch paradigm/exec)

## VERDICT (up front)
**The batch-1 short-context GEMV BW gap is STRUCTURAL on M3 Pro unified memory. Phase 2 has NO short-context GEMV tps lever beyond the already-shipped f16-scales.** Every load-width / occupancy / layout / MMA / residency form is a recorded Type-1 kill (A5/A6/A10/MMA/silicon#3). The one genuinely-distinct live formulation a careful read surfaces — a **half-precision (f16) x activation stream fed to a predec GEMV** — is **killed by byte arithmetic before any kernel work** (oracle below; x is ~0.03% of `_pair` traffic). Real forward progress is **(1) dispatch-count fusion (plan 2.2, the ~24% non-GPU gap), (2) long-context/energy via f16-KV + flash-attn, (3) the QTIP byte-cut (Phase 4).** Do **not** sink effort into lifting the predec GEMV above ~52% peak at M=1.

## (a) Why llama hits ~80% peak if simdgroup-matrix + residency are dead here
llama's `tg128` is **not** the same kernel structure as our M=1 decode GEMV. Reconciling the apparent contradiction:
- **`simdgroup matrix mul = true` is a capability flag, not proof tg128 uses MMA at M=1.** ggml-metal dispatches `mul_mv` (vec path) for M=1 and `mul_mm` (simdgroup-matrix) only for M≥`ne11`-threshold (batched/prefill). Same split dismantle already made: `Q4_K batched MMA` is **Type-1 dead at rows≤cols / N=1** (dead_levers.md "Q4_K batched MMA on rows ≤ cols", quote: *"N=1 decode wastes 7/8 of every simdgroup_multiply_accumulate; MMA is prefill-only"*; corroborated `qwen_dense.rs:4890-4896` MMA-decode exclusion). So tg128's headroom is **not** from MMA on the decode GEMV.
- **`use residency sets = true` is Type-1 dead here** (silicon #3 MTLHeap residency, quote: *"heap 8.5% slower… on unified memory the weights live in RAM and the GPU reads directly — there is no residency to optimize"*). Not the lever either.
- **Arithmetic intensity, M=1 Q4_K GEMV:** ~2 FLOP per nibble-byte streamed → ~0.5 FLOP/B → hard memory-bound; ALU is idle. dismantle's `_pair`/`_2r` (`quant.metal:2259-2267`, `:2403-2411`) are **already simdgroup-coalesced 32-lane stride-32 `uchar` loads** (A5 kill: *"loads already simdgroup-coalesced"*). So both engines are reading the same bytes the same way at M=1. The **residual gap is most plausibly (i) llama moving fewer scale bytes** (no predec table — it inline-unpacks 6-bit scales; dismantle's predec table is the +33% tax that f16-scales partially repays) **and (ii) llama's far lower dispatch count** (our 616/tok, 324 trivial). Neither is a "make the GEMV faster" lever — (i) is the byte-cut already largely banked, (ii) is plan 2.2.

## (b) Is "52% peak" the right denominator?
Partly stale. The 0.2 trace's 52%/78.7 GiB-s was measured on the **f32-predec default**, whose bytes are **~31% scale-table** (`q4k_predec_f16s_bench.rs:127`: `bytes_f32 = weights + blocks·16·4 + x + y`; for 11008×2048 gate+up: weights ≈ 25.3 MB, f32 scales ≈ 11.3 MB). **f16-scales (shipped, opt-in) removes half of that scale stream** → denominator drops ~17% (192→160 B/block, confirmed `quant.metal:2284`), so the *same µs* is a **higher % of a smaller, more-weight-dominated traffic** — post-f16s realistic ceiling is **~60-65% peak, and weights are then ~70% of traffic**. The residual after f16-scales is **bytes, not structure**: you cannot read fewer than 144 B/block of Q4_K weights. That residual is the **QTIP byte-cut's** domain, not a GEMV-kernel domain. The shipped f16-scales (+9.3% tps / −1.4% J/tok per paradigmshift.md Part II) **is the GEMV lever; it is already built** — Phase 1.2 only needs to make it a profile default, not invent new kernel work.

## (c) Any NOT-yet-killed formulation + cheap oracle?
Three candidates examined; all resolve **before** kernel work:
1. **Half-precision x stream into predec GEMV** — *genuinely distinct* from A5/A6/A10 (those touch the weight/scale load; this touches the **activation** load). A latent f16-x kernel already exists (`gemm_q4_k_m_fused_f16`, `quant.metal:1361-1419`, `device const half* x`, f32 MAC) — but it is the *non-predec* path. **KILLED BY THE OFFLINE ORACLE WITH ZERO NEW CODE:** in `q4k_predec_f16s_bench.rs:125-131`, `x_bytes = cols·4`. For the dominant `_pair` 11008×2048 shape, x = 8 KB on a ~25-37 MB read = **~0.03% of traffic**; halving x saves a rounding error. For `_2r` q/o/ffn_down (2048-4096 cols) x is still <0.1%. **Type-1 by byte math** — the denominator already counts x and it's negligible. (It *does* matter for the f32 activation buffers between dispatches — but that's plan 2.1's residual-stream/KV traffic, gated long-context, **not** the GEMV.)
2. **Dual-issue / x-vector reuse across the gate+up pair** — **already shipped.** `gemm_q4_k_v4_predec_pair` loads `xl[8]` **once** and drives both accumulator chains (`quant.metal:2255-2257`); `_2r` shares x across two rows (`:2399-2401`). The only un-fused x-reuse left is **q/k/v** (three separate `gemv_proj!` calls, no fused-QKV-pair kernel — confirmed `qwen_dense.rs:3991/4068/4079`). A fused-QKV-pair *would* reuse one x load — but x is ~0.1% of traffic (oracle #1) **and** q/k/v differ in row count/scale tables, so the win is dispatch-count (→ **plan 2.2**, not a BW lever).
3. **A fundamentally different M=1 weight layout** — **Type-1 dead** (A10 access-order repack, dead_levers.md, quote: *"repack de-coalesces (−16.8%)… the stride-32 original is already the optimally-coalesced layout"*). No live reframe (the only one — vectorized uint load on the repack — *"dies twice: NOT bit-identical… AND no BW gain"*).

**The cheap offline oracle, named:** the existing **`cargo test -p dismantle-core --test q4k_predec_f16s_bench -- --nocapture`** (GPU microbench, prints `µs/call + achieved GB/s` per shape, `:130-140`) is *the* resurrection oracle for any future GEMV-BW claim. Run it; if a proposed form does not beat the f16s GB/s readout at 11008×2048, it is dead before integration. No new tooling needed.
