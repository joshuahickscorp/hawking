# dismantle 0.1.0 — closeout

## Outcome

Two production-ready paths + one opt-in technical-curiosity path on
Apple Silicon, all parity-attested at atol < 1e-3.

## Measured perf (M3 Pro 18 GB)

| Model | Path | dec_tps median | Trials |
|---|---|---:|---|
| DeepSeek-V2-Lite Q4_K_M | indexed-no-pack-one-cb (default) | 1.61 | 3 |
| Qwen2.5-3B Q4_K_M | dense | 1.28 | 3 |
| DeepSeek-V2-Lite Q4_K_M | single-kernel (opt-in) | 0.02 | smoke |

(Honest baseline — pre-layered-MoE wedges measured 0.34 tok/s; the
batched + no-pack + one-command-buffer wedges delivered ~4.7× over that.)

## What shipped

- DeepSeek-V2-Lite MoE forward path with three layered wedges:
  batched expert GEMV, no-pack indexed expert dispatch, one-command-buffer
  MoE block. Default profile schedule: `indexed-no-pack-one-cb`.
- Qwen2.5-3B dense forward path reusing the same Metal kernel pack.
  Architecture dispatcher routes `qwen2`/`qwen2.5` GGUFs to the dense path.
- Strict single-kernel fused FlashMoE (`moe_block_fused_v2lite` +
  `moe_block_fused_v2lite_indexed`) — correctness-attested, opt-in via
  profile schedule `single-kernel`. **Not the default**: the design is
  decode-redundant (each output workgroup recomputes the full
  intermediate vector; 90× slowdown vs batched on M3 Pro decode).
- Kernel profiling + autotune system, doctor subcommand, speculative
  decoding scaffold (opt-in correctness-first).
- Eight Metal kernels parity-attested at atol=1e-3 (G1.1-G1.4 + H2.1-H2.4)
  plus two attention paths (W1B MLA gemv, W3 weight pinning) plus the
  fused MoE family (q4_one, q4_topk, v2lite, v2lite_indexed).

## What we learned

- Apple Silicon unified memory (`StorageModeShared`) makes
  `new_buffer_with_bytes` essentially free. Per-dispatch memcpy
  elimination (W3 weight pinning) delivered 0% on M3 Pro. The pinned
  surface stays in tree as a future-hardware option.
- Strict single-kernel FlashMoE in workgroup-per-output-row design is
  correct but pays redundant intermediate compute on decode mode.
  GPU concurrency doesn't recover ~250× more compute. The fix is a
  two-stage device-memory intermediate design (one command buffer,
  two dispatches: stage 1 = intermediate without redundancy, stage 2 =
  output reading the intermediate). v0.2 wedge.
- Layered MoE wedges (batched + no-pack + one-command-buffer)
  delivered the real perf (0.34 → 1.61 dec_tps, 4.7×). Dispatch
  count reduction matters; per-dispatch memcpy elimination doesn't.

## v0.2 wedges (in priority order)

1. Two-stage fused MoE (eliminate redundant intermediate compute via
   device-memory intermediate; one command buffer, two dispatches).
   Realistic target: ≥1.3× over `indexed-no-pack-one-cb`.
2. Metal MLA decode (currently CPU). Replaces `mla_decode_step`.
3. Layer-CB (extend command buffering across layer boundaries).
4. Decode arena (pool transient Metal buffer allocations).
5. fix llamacpp competitor `-ngl 99` had been done in v0.1.0; bench
   comparison vs llama.cpp Metal + MLX deferred to v0.2 with proper
   harness.
