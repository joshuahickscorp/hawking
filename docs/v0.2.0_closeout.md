# dismantle 0.2.0 — closeout

## Outcome

Four performance wedges shipped, all parity-attested at atol < 1e-3.
Default profile `v0.2.0-metal-all` activates all four simultaneously.

## What shipped

### Wedge 1 — Metal MLA decode
Replaces the CPU `mla_decode_step` with `mla_decode_kernel` (Metal).
Operates on the compressed KV representation (c_kv, k_pe) for the full
sequence in one dispatch. Active when `mla_schedule == "metal-mla"`.
Files: `shaders/attn.metal`, `kernels/mod.rs`, `deepseek_v2.rs`.

### Wedge 2 — Two-stage fused MoE
Replaces the single-kernel workgroup-per-output-row FlashMoE (which paid
redundant intermediate compute on every output row) with a two-kernel
design:
- Stage 1 `moe_block_two_stage_intermediate`: one workgroup per expert
  slot × mid row, writes all expert intermediates to device memory.
- Stage 2 `moe_block_two_stage_output`: one workgroup per output row,
  reads shared device-memory intermediates, writes weighted sum.
Both stages run in one command buffer via `dispatch_batch`. Active when
`moe_schedule == "two-stage"`.
Files: `shaders/moe.metal`, `kernels/mod.rs`, `deepseek_v2.rs`,
`tests/phase2_moe_block_fused_parity.rs` (`test_moe_block_two_stage_matches_cpu`).

### Wedge 3 — Layer command-buffer batching (Layer-CB)
Batches `mla_decode_kernel` + `gemv_f32_attn` (o_proj) into a single
command buffer via `dispatch_batch`. Saves one `commit` + `wait_until_completed`
per attention layer = 27 fewer GPU round-trips per token on
DeepSeek-V2-Lite. Active when `command_buffering == "layer-cb"`.
Files: `kernels/mod.rs` (`mla_decode_and_o_proj_metal`), `deepseek_v2.rs`.

### Wedge 4 — Decode-arena Metal buffer pool
Pre-allocates five Metal buffers (q, c_kv, k_pe, attn_out, out) once at
load time, reuses them across all decode steps. Eliminates per-dispatch
`new_buffer`/`new_buffer_with_bytes` calls in the MLA attention hot path.
When active, writes CPU data into the shared buffers via plain `memcpy`
(no allocation), then dispatches `mla_decode_and_o_proj_arena_metal`.
Active when `gpu_buffer_reuse == "decode-arena"`.
Files: `metal/decode_arena.rs` (new), `metal/mod.rs`, `kernels/mod.rs`
(`mla_decode_and_o_proj_arena_metal`), `deepseek_v2.rs`.

### Profile system
Added `v0.2.0-metal-all` kernel variant (score 175, highest ranked) that
activates all four wedges. `deterministic_rank: 5` ensures it wins
`select_variant` in the autotune scoring pass.
Files: `profile.rs`.

## Measured perf (M3 Pro 18 GB)

Bench measured in a slm-quiet window, 3 trials × 64 tokens each, "Once upon
a time" prompt, greedy temp=0.

| Path | dec\_tps | vs v0.1.2 baseline | trials |
|---|---:|---:|---|
| v0.2.0-metal-all (all four wedges) | **0.57** | **0.36×** (regression) | 0.550, 0.566, 0.568 |
| v0.1.2 default (indexed-no-pack-one-cb) | **1.59** | 1.0× (baseline) | 1.584, 1.586, 1.612 |

**⚠️ Regression finding:** v0.2.0-metal-all is 2.8× slower than the v0.1.2 default.
The v0.1.2 default schedule remains the recommended production path.

## Regression root cause

Metal MLA decode (Wedge 1) adds **27 GPU roundtrips per token** for attention (one
`commit`+`wait_until_completed` per layer even with Layer-CB). The v0.1.2 path uses
the CPU for MLA computation with **zero** GPU roundtrips for attention. At the
tested sequence length (64 tokens), GPU kernel launch overhead dominates over the
attention arithmetic benefit.

Simultaneously, Two-stage MoE (Wedge 2) is slower than `indexed-no-pack-one-cb` at
short sequence decode because it runs two kernel dispatches per expert block instead
of the batched one-command-buffer path that was already optimized in v0.1.2.

Root cause summary:
- Metal MLA: GPU launch overhead > arithmetic savings at ≤64 token sequences.
- Two-stage MoE: `dispatch_batch` with intermediate synchronization point slower than
  the existing optimized `indexed-no-pack-one-cb` at decode batch size 1.
- Layer-CB and decode-arena (Wedges 3+4): correct engineering, but offset by the above.

## Fix plan for v0.2.1

1. **Metal MLA threshold gate**: Only activate Metal MLA when `seq_len > N` (TBD via
   bisect bench). Below threshold, fall back to CPU MLA. N is likely 128-512.
2. **Two-stage MoE vs indexed-no-pack comparison**: Per-wedge bench to confirm
   two-stage is the bottleneck; may need to revert MoE schedule back to
   `indexed-no-pack-one-cb` as default and only use `two-stage` for fill contexts.
3. **GPU roundtrip profiling**: Use Metal capture to attribute exact time per
   attention layer dispatch vs MoE dispatch; fix the dominant term first.

## What we learned

- `dispatch_batch` (one command buffer, multiple kernels) is the right
  abstraction for multi-stage kernels. Metal guarantees in-order execution
  within a single command buffer, so no explicit synchronization is needed
  between stages.
- The two-stage design for MoE (device-memory intermediate) is the correct
  fix for the single-kernel decode-redundant compute finding from v0.1.0.
- `StorageModeShared` `write_buffer_bytes` (memcpy into pre-allocated buffer)
  vs `new_buffer_with_bytes` (allocate + memcpy): the allocation savings are
  real but may be small on Apple Silicon unified memory, consistent with the
  v0.1.0 observation that per-dispatch memcpy is near-free.
- Layer-CB dispatch reduction (27 fewer round-trips) is the most
  mechanically impactful change; impact depends on how much of decode time
  is CPU-GPU synchronization vs actual kernel execution.
