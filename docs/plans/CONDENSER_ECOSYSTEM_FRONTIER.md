# HAWKING CONDENSER ECOSYSTEM FRONTIER

## Canonical constitution for making an extreme condensed model the best local agent substrate, not merely the smallest model that runs

**Status:** Proposed additive successor constitution for the post-proof Hawking product and ecosystem layer  
**Date:** 2026-07-17  
**Primary model objective:** Preserve the `EXTREME` Event Horizon target: the lowest exact physical whole-artifact BPW that passes the frozen standalone-model contract  
**Primary ecosystem objective:** Maximize verified local work completed per installed byte, resident byte, context byte, user intervention, and wall-clock unit  
**Recommended user profile:** `EXTREME_LOCAL`  
**Activation boundary:** Additive only. It may define new wiring, profiles, APIs, state objects, context policies, and UX surfaces, but it may not mutate, relabel, interrupt, or retroactively reinterpret the active Doctor result campaign  
**Speed policy:** Raw serving speed remains a separate optimization and receipt. Context, continuity, tooling, interoperability, and agent usability are product requirements even when no speculative-decoding or throughput work is performed

> **Press makes the model small. Doctor makes it a model again. Context lets it see enough. Continuum lets it remember. Bridge lets the ecosystem use it. HIDE makes the machinery feel like magic.**

## 0. Paste target

Create and pursue a successor goal named **HAWKING CONDENSER ECOSYSTEM FRONTIER**.

Do not change the scientific identity, scheduling, observations, artifacts, progress, or terminal criteria of the current condensation ladder. Treat completed and running Doctor cells as immutable evidence. This generation changes how proven artifacts are **wired, summoned, contextualized, resumed, exposed, and experienced** after they exist.

Begin from the recommended extreme target:

```text
EXTREME
    = the lowest exact physical whole-artifact BPW that passes the frozen
      standalone production-quality contract.

EXTREME_LOCAL
    = that EXTREME artifact, combined with the strongest independently proven
      local context, state, tool, safety, portability, and UX profile that fits
      the declared device envelope without borrowing capability from a hidden
      remote model.
```

The purpose is not to make Hawking another chat window, another API wrapper, another IDE extension, or another generic local inference server.

The purpose is to make a Hawking-condensed model **more desirable to use than an ordinary quantized copy of the same or smaller parent** because the Hawking artifact can deliver:

- more parent capability inside the same device;
- more validated context inside the same memory envelope;
- persistent and resumable local state;
- better structured-output and tool-use reliability;
- a device-aware context and memory policy;
- progressive quality slices;
- exact capability and context receipts;
- open API and MCP interoperability;
- private, offline operation;
- a reference agent experience that does not lock the artifact inside one application; and
- an evidence-backed explanation of why this is the correct artifact and profile for the user’s machine.

The final result is not:

> “The Hawking model is smaller than the GGUF.”

The final result is:

> “On this exact machine, this Hawking artifact fits a stronger parent, holds a longer verified working context, survives long agent sessions, resumes exactly after interruption, connects to the tools and clients the user already has, and completes more useful local work than the best ordinary quantized alternative under the same total device budget.”

## 1. The new thesis

Hawking began with a model-size question:

> How far can the weights be condensed before the model stops satisfying a capability contract?

The ecosystem frontier adds a deployment question:

> After Hawking finds that model boundary, how much **local agency** can be built around the artifact before memory, context reliability, continuity, tool reliability, or user attention becomes the next binding limit?

A user does not experience BPW directly. The user experiences whether the system can:

- load the model;
- understand the workspace;
- retain the right information;
- use tools correctly;
- continue a task for hours or days;
- recover after a crash or reboot;
- remain private and offline;
- explain its limits;
- work with existing clients; and
- complete the task without the user repeatedly rebuilding context.

Therefore Hawking’s full product is not only a codec or runtime. It is a **condenser operating layer** for local intelligence.

The condenser mindset applies to every scarce resource:

```text
weights          → remove redundant model representation
Doctor bytes     → add only causally necessary repair
KV state         → retain the highest-value exact or compressed state
prompt context   → compile rather than concatenate
workspace data   → index and retrieve rather than stuff
agent history    → checkpoint, summarize, and branch rather than replay blindly
tool schemas     → defer and summon rather than preload everything
model variants   → progressive slices rather than duplicate full artifacts
repository code  → a small active core that summons pinned packs
user attention   → ask only for decisions that cannot be made safely from evidence
```

Hawking should condense the **entire local intelligence stack**, while refusing to compress away provenance, safety gates, rollback, or proof.

## 2. The recommended result

### 2.1 `EXTREME` remains the recommended model artifact

No ecosystem feature may weaken the Event Horizon model contract.

`EXTREME` remains:

```text
the smallest exact standalone artifact that passes all frozen model-quality,
safety, packaging, deterministic-load, and reproducibility requirements.
```

Retrieval, tools, external models, runtime memories, and agent scaffolding may not be used to rescue a failed standalone-model claim.

### 2.2 `EXTREME_LOCAL` becomes the recommended way to use it

`EXTREME_LOCAL` is not a different model-quality claim. It is a deployment profile bound to the sealed artifact.

It selects:

- the largest **validated effective context** that fits;
- the correct position policy;
- the correct KV representation and retention policy;
- the correct persistent-state tier;
- the correct workspace index and retrieval budget;
- the correct tool grammar and permission profile;
- the correct agent/session checkpoint policy;
- the correct progressive artifact slices;
- the correct local API surfaces; and
- the correct fallbacks when a requested feature exceeds evidence.

The default product recommendation is therefore:

```text
recommended_model      = EXTREME
recommended_experience = EXTREME_LOCAL
```

### 2.3 Balanced and fidelity remain visible

Hawking should still expose:

- `BALANCED_LOCAL`: the knee across quality, context, memory movement, and responsiveness;
- `FIDELITY_LOCAL`: the strictest parent-relative artifact plus the strongest validated local context; and
- `ARCHIVE_LOCAL`: the smallest durable artifact that preserves a declared archival capability contract and exact lineage.

But the product should lead with `EXTREME_LOCAL`, because the point of Hawking is to move the physical boundary.

## 3. Claim separation

The ecosystem frontier introduces useful runtime capabilities without allowing them to contaminate the model claim.

Every result must identify one of these layers:

1. **Standalone artifact capability**  
   What the sealed Hawking model can do with no retrieval, tools, external model, or persistent information beyond the declared prompt.

2. **Context-system capability**  
   What the same artifact can do with a declared local context policy, KV codec, retrieval index, state store, or recurrent memory. All state bytes and preprocessing are billed.

3. **Agent-system capability**  
   What the artifact can do with declared tools, permissions, orchestration, subagents, verification, and session state. All calls, state, retries, and user interventions are billed.

4. **External augmented capability**  
   What the system can do with remote search, cloud tools, or another model. This remains a separate augmented-system claim and can never support a local standalone headline.

A model can fail the standalone Event Horizon and still participate in a useful augmented research experiment, but Hawking must never rename that system result as model restoration.

## 4. Mathematical constitution

### 4.1 Total local memory is the real device contract

For artifact `a`, context policy `c`, session state `s`, workspace `w`, and runtime `r`:

\[
M_{local}(a,c,s,w,r)
=
M_{artifact}(a)
+M_{resident\ expansion}(a,r)
+M_{KV}(a,c)
+M_{recurrent}(a,c)
+M_{session}(s)
+M_{index}(w)
+M_{workspace\ cache}(w)
+M_{runtime}(r).
\]

The device admission rule is:

\[
M_{local}^{worst}
+M_{OS\ reserve}
+M_{safety\ reserve}
\le M_{device}.
\]

An ordinary quant comparison must use the same total-memory equation. It is not enough to compare only model files while letting one system consume unlimited KV, retrieval, cache, or helper-model memory.

### 4.2 KV memory

For a transformer with `L` layers, `H_kv` KV heads, head dimension `d_h`, KV precision `q` bits, batch or slot count `B`, and context length `C`:

\[
M_{KV}(C)
=
B \cdot C \cdot
2L H_{kv} d_h \frac{q}{8}
+M_{KV\ metadata}.
\]

The two is for keys and values.

The maximum context allowed by a remaining memory budget is approximately:

\[
C_{fit}
=
\left\lfloor
\frac{M_{device}-M_{fixed}}
{B \cdot 2L H_{kv} d_h(q/8)}
\right\rfloor,
\]

where `M_fixed` includes the artifact, resident decoder structures, workspaces, OS reserve, session state, and safety margin.

This equation makes the connection between condensation and context explicit: every real byte removed from the model can become KV or persistent-state headroom.

### 4.3 The Hawking context dividend

Let `M_Q` be the complete fixed memory of the strongest ordinary quantized competitor and `M_H` the complete fixed memory of the Hawking artifact, including Doctor treatment and resident expansion.

\[
\Delta M_{context}=M_Q-M_H.
\]

For a dense full-KV policy:

\[
\Delta C_{raw}
=
\left\lfloor
\frac{\Delta M_{context}}
{B \cdot 2L H_{kv}d_h(q/8)}
\right\rfloor.
\]

This is a **capacity dividend**, not a quality claim. Hawking earns it only when the extended context passes its own evaluation contract.

### 4.4 Context Horizon

A nominal token limit is not a context result.

For context length `C`, define a protected context-capability vector:

\[
Q_{ctx}(C)
=
\left(
Q_{retrieval},
Q_{multihop},
Q_{instruction},
Q_{ordering},
Q_{code},
Q_{state},
Q_{calibration}
\right).
\]

The **Context Horizon** is:

\[
C_H(a,d,\Gamma_{ctx})
=
\sup\left\{
C:
M_{local}(C)\le M_d
\land
\forall j,\
\operatorname{LCB}(Q_{ctx,j}(C))\ge \tau_j
\right\}.
\]

Hawking reports the largest measured passing rung and the next unresolved or failing rung.

### 4.5 Effective context

`C_nominal` is the number the runtime accepts. `C_effective` is the amount of information the system can reliably use.

A conservative summary is:

\[
C_{effective}(C)
=
C\cdot
\min_j R_j(C),
\]

where `R_j` are normalized retention factors for retrieval, positional stability, multi-hop composition, instruction persistence, code-reference accuracy, and state reconstruction.

This scalar is only a summary. The full vector and per-depth curves remain authoritative.

### 4.6 Continuity Horizon

An agent session is useful only if it can survive interruption without losing intent or silently changing state.

Let:

- `p_checkpoint` be the probability a durable checkpoint is complete and valid;
- `p_replay` be the probability the event/tool ledger reconstructs the same logical state;
- `p_context` be the probability context compilation produces an equivalent admissible working set;
- `p_artifact` be the probability the same artifact/profile identity is restored; and
- `p_task` be the probability the resumed agent remains on the same verified task trajectory.

Then:

\[
P_{continuity}
=
p_{checkpoint}
\cdot p_{replay}
\cdot p_{context}
\cdot p_{artifact}
\cdot p_{task}.
\]

The **Continuity Horizon** is the longest measured session duration, event count, or task depth at which the lower confidence bound of `P_continuity` remains above the frozen threshold.

### 4.7 Autonomy Horizon

For an agent policy `\pi`, workload distribution `\mathcal D`, and maximum number of model/tool transitions `n`:

\[
A_H
=
\sup\left\{
n:
\operatorname{LCB}
\left(
\mathbb E_{x\sim\mathcal D}[U(x,\pi,n)]
\right)
\ge \tau_U,
\quad
H_{intervention}(n)\le H_{max}
\right\}.
\]

The user should see both:

- the longest reliable unattended or low-intervention task depth; and
- the failure mode that appears beyond it.

### 4.8 Local agency density

The ecosystem objective is not tokens per second. It is verified useful work per complete local cost:

\[
\eta_{agency}
=
\frac{
\sum_i U_i\cdot I_i^{verified}
}{
\alpha B_{installed}
+\beta B_{resident\cdot time}
+\gamma B_{moved}
+\delta E
+\epsilon T
+\zeta H_{human}
}.
\]

Where:

- `U_i` is task utility;
- `I_i^verified` is one only when the task outcome passes the declared verifier;
- `B_installed` includes the artifact, optional slices, indexes, and model-specific persistent state;
- `B_resident·time` charges memory occupation;
- `B_moved` charges memory and storage traffic;
- `E` is energy;
- `T` is wall time; and
- `H_human` is user intervention.

Weights are published rather than hidden inside one proprietary score.

### 4.9 Ecosystem Pareto frontier

For a complete local configuration `z = (artifact, context, state, tools, UX profile)`, maintain the nondominated set over:

\[
F(z)=
\left(
Q_{model},
R_{physical},
C_H,
P_{continuity},
A_H,
\eta_{agency},
M_{resident},
B_{moved},
T_{task},
H_{human},
P_{privacy},
P_{interop},
P_{proof}
\right).
\]

Hawking should not collapse this into one score for scientific claims. Product profiles may choose points from the frontier under transparent preferences.

### 4.10 Equal-device superiority test

A Hawking configuration is preferable to an ordinary quantized competitor only when the comparison freezes:

- the exact device and safety reserve;
- total installed and resident bytes;
- context-quality gates;
- tool catalog and tool budgets;
- workspace/index bytes;
- test-time compute;
- user intervention policy;
- privacy/network policy;
- task suite; and
- proof level.

Hawking wins a user-facing comparison when it is no worse on every frozen requirement and strictly better on at least one of:

- verified task completion;
- parent capability;
- effective context;
- continuity;
- local privacy;
- installed size;
- resident memory;
- user intervention; or
- supported device class.

## 5. Extend the condenser mindset beyond weights

### 5.1 Condense the representation

Use the Event Horizon and Doctor to minimize the complete standalone artifact.

### 5.2 Condense context, not truth

A context compiler should select the smallest sufficient causal working set rather than paste every available token.

It may compress:

- duplicate code spans;
- repeated tool output;
- already-proven facts;
- stale diagnostics;
- superseded plans;
- redundant document chunks;
- repeated system instructions; and
- old conversational turns whose durable state has been extracted.

It must not compress away:

- current user intent;
- unresolved constraints;
- safety and permission state;
- provenance;
- exact identifiers needed for action;
- failed attempts that affect the next decision;
- active branch or artifact identity; or
- information required to reproduce a tool action.

### 5.3 Condense state through typed objects

Agent state should be stored as typed, content-addressed objects instead of one ever-growing transcript:

```text
TaskIntent
PlanGraph
WorkspaceSnapshot
ArtifactIdentity
ContextManifest
ToolLedger
PermissionLedger
OpenProcessSet
EvaluationState
MemoryFact
UncertaintyRegister
Checkpoint
```

The prompt is a compiled view over these objects, not the canonical state itself.

### 5.4 Condense tool exposure

Do not inject every tool schema into every turn.

Use:

- deferred tool loading;
- tool search;
- namespace summaries;
- task-specific allowlists;
- compact examples;
- schema caching; and
- programmatic tool execution where intermediate data can remain outside the model context.

### 5.5 Condense model variants

A progressive Hawking artifact should expose verified slices rather than force users to store unrelated copies:

```text
base extreme slice
+ context-rehabilitation slice
+ coding/tool-use slice
+ fidelity slice
+ architecture-specific optional slice
```

Each slice has exact bytes, dependencies, capability effects, and a separate signature.

### 5.6 Condense user attention

The advisor should make every safe, evidence-supported decision automatically.

Ask the user only when:

- two frontier points encode a real preference tradeoff;
- a permission or destructive action requires consent;
- licensing blocks automatic redistribution;
- the requested context exceeds the validated horizon; or
- no evidence-supported profile exists.

## 6. The Context Frontier

### 6.1 Context is a model-quality problem and a systems problem

Long context fails for different reasons:

- the model was not trained for the position range;
- RoPE extrapolation drifts;
- attention loses important earlier tokens;
- KV quantization error compounds;
- eviction removes a causal dependency;
- the prompt compiler selects the wrong evidence;
- the model suffers lost-in-the-middle behavior;
- the workspace index misses a required symbol;
- a summary removes a constraint;
- a recurrent state saturates;
- a retrieval system returns stale or contaminated information; or
- the artifact’s extreme compression damaged long-range behavior specifically.

Hawking must diagnose these separately.

### 6.2 Weight compression creates context headroom

The first local-context advantage is direct:

```text
smaller healed weights
    → more unified memory left for KV, recurrent state, indexes, and workspaces
    → longer or richer local sessions on the same machine
```

This is a major reason to prefer a Hawking artifact over an ordinary quant even when both produce similar short-context benchmark scores.

The comparison must include the Doctor bytes and any resident decoding expansion. Hawking earns only real headroom.

### 6.3 Position extension

For RoPE models, Hawking may support:

- native positions;
- positional interpolation;
- Dynamic NTK;
- YaRN;
- model-specific learned scaling;
- long-context fine-tuning; and
- position-aware Doctor rehabilitation.

Every target context rung must pass actual long-context tasks. “It loaded” and “it generated fluent text” are not context proof.

### 6.4 KV precision and allocation

KV should be allocated by sensitivity rather than one global dtype.

Candidate policies include:

- native/F16 control;
- per-channel INT8;
- per-channel INT4;
- INT2 where evidence permits;
- mixed precision by layer/head/state age;
- higher precision for attention sinks and recent tokens;
- higher precision for retrieval-critical heads; and
- codec-native correction for quantization-sensitive layers.

Doctor should build a KV damage atlas and allocate the minimum bytes that preserve the context contract.

### 6.5 Tiered state: STKV as a Hawking-specific product

The target state hierarchy is:

```text
HOT
    exact or high-precision attention sinks, current tool state, and recent window

WARM
    compressed but addressable KV pages selected by measured attention/retrieval value

COLD-EXACT
    lossless pages on SSD, content-addressed and fetched on demand

COLD-SUMMARY
    recurrent/SSM state or verified structured summaries when exact token recall is not required

ARCHIVE
    immutable session/event objects from which a new context view can be reconstructed
```

A policy chooses between exact paging and lossy summarization based on the task contract.

The system must never say “unbounded exact context” when the cold tail is summarized.

### 6.6 Dense attention and recurrent models are complementary

RWKV/Mamba-style recurrent state offers flat memory with context length, but fixed state does not guarantee exact recall of arbitrary old details.

Hawking should combine:

- recurrent state for continuous semantic continuity;
- exact local attention for the current working set;
- sparse/indexed retrieval for old exact facts;
- SSD pages for auditable exact history; and
- Doctor-trained interfaces between those state types.

The product objective is not ideological purity. It is the smallest state system that passes the workload’s recall and continuity contract.

### 6.7 Context compiler

The compiler is a constrained selection problem.

For candidate context item `i`, define:

\[
V_i
=
w_r R_i+w_c C_i+w_t T_i+w_p P_i-w_d D_i-w_s S_i,
\]

where:

- `R_i` is relevance;
- `C_i` is causal importance;
- `T_i` is recency;
- `P_i` is provenance/priority;
- `D_i` is redundancy; and
- `S_i` is staleness or contamination risk.

Select items under region, ordering, and token constraints:

\[
\max_x \sum_i x_iV_i
\quad
\text{subject to}
\quad
\sum_i x_it_i\le C_{budget},
\]

plus mandatory reservations for:

- system and safety policy;
- current intent;
- output;
- working scratch;
- active files/symbols;
- active tool state;
- unresolved failures; and
- recent user-visible decisions.

The compiler should use head/tail placement, deduplication, structural grouping, and iterative improvement. It should emit a signed `ContextManifest` explaining every included and excluded object.

### 6.8 Workspace Lens

For coding and research agents, the context system needs a local workspace intelligence layer:

- content hashes and Merkle identity;
- tree-sitter or equivalent syntax structure;
- symbol definitions and references;
- dependency and call graphs;
- diagnostics;
- semantic and lexical retrieval;
- recency and modification state;
- repository map under a token budget;
- changed-file and active-task boosts;
- generated/vendor exclusions; and
- exact links back to source ranges.

The agent should receive the smallest high-value map, not the entire repository.

### 6.9 Prefix and branch DAG

Repeated agent work should reuse exact state.

Represent prefixes and branches as immutable content-addressed nodes:

```text
parent state hash
+ token or event delta
+ model/tokenizer/profile identity
+ position policy
+ KV/state codec
= child state identity
```

This enables:

- exact prefix reuse;
- copy-on-write branches;
- session forking;
- rollback;
- subagent context firewalls;
- cache deduplication;
- reproducible evaluation; and
- portable session capsules.

### 6.10 Memory as provenance-bound local state

Long-running agents need memory, but memory must not become an unreviewed source of false authority.

Every memory object records:

- content;
- source event or file;
- author/actor;
- confidence;
- creation and last-validation time;
- workspace/model scope;
- privacy class;
- expiration or revalidation policy;
- contradiction links; and
- whether it may affect planning, action, or only retrieval.

The memory system must support correction and forgetting. Persistent does not mean permanently trusted.

### 6.11 Context evaluation battery

Every promoted context profile should cover:

- needle retrieval at multiple depths and densities;
- multi-needle binding;
- multi-hop reasoning across distant evidence;
- lost-in-the-middle tests;
- instruction persistence;
- long code navigation and refactoring;
- long agent trace continuation;
- repeated tool-output compression;
- stale-memory rejection;
- branch/fork isolation;
- resume after process exit and reboot;
- KV/state codec ablations;
- position-policy ablations;
- worst-layer/head sensitivity; and
- context-length calibration.

Report quality as a curve over length, depth, state age, and task class.

## 7. Context Doctor

Doctor should expand from treating weight damage to treating the model’s **context phenotype**.

### 7.1 Context pathology classes

1. `position_drift`  
   The model’s representation degrades as positions exceed training support.

2. `lost_middle`  
   Relevant information survives in the input but is not used reliably at middle positions.

3. `kv_quant_damage`  
   Lower-precision state changes attention or outputs beyond the context contract.

4. `state_age_decay`  
   Compressed or recurrent state becomes inaccurate with age.

5. `eviction_causal_loss`  
   A retention policy removes information later required for the task.

6. `retrieval_selection_failure`  
   The correct evidence exists locally but is not selected.

7. `summary_constraint_loss`  
   A compactor preserves topic but removes a binding instruction, identifier, or failure.

8. `workspace_map_failure`  
   The agent misses a symbol, dependency, diagnostic, or changed file.

9. `tool_trace_amnesia`  
   The model or context compiler loses what actions were already performed and why.

10. `compression_context_interaction`  
    The extreme weight artifact retains short-context capability but loses long-context behavior disproportionately.

### 7.2 Context treatment families

Doctor may prescribe:

- position scaling or long-context fine-tuning;
- position-aware QAT;
- long-sequence parent distillation;
- protected attention layers or heads;
- mixed-precision KV allocation;
- attention-sink preservation;
- retrieval-critical state islands;
- learned KV correction;
- exact-page fallback;
- context compiler weight changes;
- structured summary schemas;
- memory revalidation;
- workspace-index repair;
- state reset or re-encoding; and
- closing the requested context as unsupported.

### 7.3 Hawking-native context mechanisms

#### Context Horizon Waterfilling (`CHW`)

Jointly allocate model bits, Doctor bytes, KV precision, exact state pages, index bytes, and context tokens under one device-memory ceiling.

#### Positional Rehabilitation Field (`PRF`)

Learn a compact position-dependent correction or base rewrite that restores hidden/logit geometry across extended positions without shipping a dense shadow.

#### KV Immunization Map (`KVIM`)

Measure layer/head/state-age sensitivity and allocate higher state precision only where lower precision causally damages protected context tasks.

#### Causal Memory Islands (`CMI`)

Identify the smallest exact historical spans that must remain lossless because later actions depend on them.

#### Semantic State Codec (`SSC`)

Encode older state into typed semantic objects with explicit uncertainty, provenance, and exact-page pointers.

#### Context Error Propagation Firewall (`CEPF`)

Detect early positional or state errors that amplify across later layers and protect their source components.

#### Context Treatment Transplant (`CTT`)

Test whether a context treatment learned on one scale, architecture, or task transfers as a scheduling prior. It never substitutes for target-specific evidence.

#### Context Autopsy Ledger (`CAL`)

Retain failed context lengths, KV policies, position methods, compiler settings, and exact failure signatures so the same dead route is not repeatedly rediscovered.

### 7.4 Context Doctor claim boundary

A context treatment embedded in the standalone artifact may contribute to the artifact’s physical BPW and standalone context claim.

A runtime index, external memory, tool trace, or SSD state plane contributes only to the context-system or agent-system claim. Doctor must preserve the distinction.

## 8. The Agent Frontier

### 8.1 The agent should be a deterministic shell around a probabilistic model

The reference loop should be an explicit state machine:

```text
INTAKE
→ ORIENT
→ COMPILE_CONTEXT
→ PLAN
→ REQUEST_PERMISSION
→ ACT
→ OBSERVE
→ VERIFY
→ REPAIR | REPLAN | COMPLETE | PAUSE
→ CHECKPOINT
```

No state advances merely because the model says it succeeded.

### 8.2 Tool surface

Hawking should provide a small, sharp built-in tool set and use MCP for breadth.

Core local tools should cover:

- file read/search/write/edit;
- line-number-free patch application;
- shell/process execution and monitoring;
- repository status, diff, commit, branch, and worktree operations;
- diagnostics and language-server queries;
- structured task/plan state;
- image/document viewing where supported;
- local web/document retrieval under policy;
- permission requests;
- checkpoints and session branches; and
- MCP tool/resource/prompt discovery.

Each tool is a contract designed for model ergonomics:

- concise namespaced names;
- strict schemas;
- high-signal results;
- bounded output with pagination/spill;
- actionable errors;
- idempotency keys;
- dry-run/simulation where meaningful;
- exact side-effect receipts; and
- permission classification.

### 8.3 Structured-output reliability

Tool calls, patches, plans, and machine-consumed outputs should use grammar- or schema-constrained decoding.

The requirements are:

- invalid tokens masked before sampling;
- schema identity cached;
- free-text reasoning separated from constrained action fields;
- batched and single-sequence paths behaviorally aligned;
- streaming arguments validated before irreversible execution;
- deterministic parser fallbacks;
- exact tool-selection constraints; and
- conformance measured, not assumed.

Doctor should treat structured-output and tool-use degradation as capability failures of the condensed model.

### 8.4 Agent Grammar Rehabilitation (`AGR`)

Measure whether condensation disproportionately harms:

- JSON/schema compliance;
- tool selection;
- argument binding;
- edit format;
- stop behavior;
- retry behavior;
- plan stability; and
- permission obedience.

Treatments may include:

- capability-targeted correction bases;
- parent distillation on tool traces;
- grammar-aware QAT;
- protected output-head/lexical regions;
- structured failure replay;
- edit-specific rehabilitation; and
- closing unsupported tool modes.

### 8.5 Permission and safety UX

The user should not have to choose between total automation and constant interruption.

Use layered policies:

```text
read-only
workspace-write
network-restricted
system-change
full-access
```

Each call declares:

- requested capability;
- exact target;
- expected side effect;
- reversible or irreversible status;
- data leaving the device;
- command or patch preview;
- reason; and
- whether a narrower safe alternative exists.

The permission engine, not the model, is authoritative.

### 8.6 Event-sourced sessions

Every agent action and observation should enter a causal event log with:

- stable event id;
- sequence;
- timestamp;
- actor;
- cause/parent event;
- task and branch identity;
- tool/model identity;
- input/output hashes;
- permission receipt;
- result status;
- context manifest;
- artifact/profile identity; and
- chain integrity.

This supports replay, forking, audit, crash recovery, evaluation, and UI time travel.

### 8.7 Agent Continuity Capsule (`ACC`)

A portable session capsule includes:

- task intent;
- current plan DAG;
- event-log head;
- active artifact and profile;
- workspace root and snapshot identity;
- context object manifests;
- memory objects;
- open process handles that can be safely reattached or marked dead;
- tool and permission ledgers;
- pending approvals;
- evaluation/verifier state;
- exact resume command; and
- migration schema.

The capsule should survive process exit, application restart, and machine reboot.

### 8.8 Subagents as context firewalls

Subagents should receive:

- a narrow task;
- a bounded context manifest;
- a tool allowlist;
- a model/profile selection;
- a budget;
- a separate event branch; and
- an explicit return schema.

Only verified results and selected evidence return to the parent. Full child transcripts do not automatically pollute the parent context.

### 8.9 Verification-first UX

The UI should distinguish:

- model output;
- proposed action;
- tool execution;
- observed result;
- verifier verdict;
- unresolved uncertainty; and
- user-approved claim.

A green check means a verifier passed, not that the model sounded confident.

### 8.10 Agent evaluation battery

Measure:

- end-to-end verified task completion;
- tool-call schema validity;
- tool selection accuracy;
- argument correctness;
- edit application success;
- compilation/test pass rate;
- permission compliance;
- rollback success;
- crash-resume equivalence;
- context rebuild equivalence;
- redundant tool calls;
- user interventions;
- branch contamination;
- long-session drift;
- memory correction/forgetting;
- offline completion; and
- equal-device performance against ordinary quantized alternatives.

## 9. The UX Frontier

### 9.1 The magic command

The ideal experience is:

```sh
hawking use <model-or-capsule> \
  --intent coding-agent \
  --workspace . \
  --profile extreme-local
```

Hawking should then:

1. verify the artifact and Passport;
2. inspect the device and free resources;
3. select the validated model/context/state profile;
4. build or reuse the workspace Lens;
5. expose the local APIs;
6. attach HIDE or another client;
7. restore the last valid session if requested;
8. show the real capability and context envelope; and
9. refuse unsupported context or tool modes honestly.

### 9.2 Device and intent advisor

The advisor accepts:

- machine;
- model/capsule;
- intended workload;
- desired privacy;
- maximum disk and memory;
- desired context;
- responsiveness preference;
- tool permissions; and
- whether network access is allowed.

It returns a ranked frontier, with `EXTREME_LOCAL` first when admissible.

### 9.3 Context dial with a warranty

The user should see:

```text
Native        32K    strongest validated fidelity
Long         128K    passes retrieval/code/agent gates
Frontier     256K    passes most gates; one declared weak slice
Stream       unbound bounded-state conversation; not exact full recall
```

The dial must show:

- nominal tokens;
- effective held context;
- KV/state bytes;
- memory pressure margin;
- position method;
- retention policy;
- exact versus summarized history;
- measured quality gates; and
- expected failure mode.

### 9.4 Progressive artifact UX

The user can begin with the extreme slice and add signed improvements:

```text
model.hawking/
  core.extreme
  slice.context-128k
  slice.coding-tools
  slice.fidelity
  slice.multilingual-tail
  passport.json
```

The runtime activates only compatible, hash-bound slices. Removing a slice returns to a known profile.

### 9.5 Session UX

The reference interface should support:

- pause;
- resume;
- fork;
- rewind to an event;
- compare branches;
- inspect context composition;
- inspect memory and provenance;
- inspect Doctor treatments;
- change profile at a safe checkpoint;
- export a session capsule; and
- continue from another compatible Hawking installation.

### 9.6 Model Surgery view

Users and researchers should be able to inspect:

- base representation by tensor group;
- protected precision islands;
- Doctor treatment bytes;
- capability changes;
- context-specific treatments;
- known weak domains;
- proof state;
- exact artifact size; and
- comparison with ordinary quantized controls.

This turns “magic” into understandable engineering without making ordinary use complicated.

### 9.7 Offline-first and local-network UX

After artifacts, indexes, and tools are installed, core operation should require no network.

The user may opt into:

- localhost only;
- authenticated local-network service;
- remote MCP servers;
- remote search; or
- distributed local devices.

Every transition that sends data off-device is visible and policy-controlled.

### 9.8 HIDE is the reference shell, not the prison

HIDE should demonstrate the complete Hawking experience:

- device advisor;
- model/session management;
- agent loop;
- context inspection;
- tool permissions;
- verification;
- branchable event history;
- Doctor and Passport views; and
- local workspace intelligence.

But every core capability must remain available through APIs and CLI so users can choose another IDE, agent, notebook, service, or UI.

## 10. Open ecosystem contract

### 10.1 Inputs

Hawking should accept, where architecture and license permit:

- Hugging Face repositories and `safetensors`;
- GGUF sources;
- model-specific native formats;
- local files and verified remote ranges;
- tokenizer/chat-template assets;
- adapters and deltas with explicit provenance; and
- source manifests for bring-your-own-source derivation.

### 10.2 Outputs

Primary output:

- a native `.hawking` capsule supporting extreme codecs, Doctor treatments, progressive slices, state contracts, and proof receipts.

Compatibility outputs, where semantically possible:

- GGUF or another standard quant artifact;
- `safetensors` reconstruction oracle for research only;
- model metadata and tokenizer bundles;
- portable Passport JSON;
- runtime adapter packages; and
- derivation recipes for licenses that forbid redistribution.

A standard export may lose Hawking-native features. The export receipt must say exactly what was dropped or materialized.

### 10.3 API compatibility

Hawking should support:

- OpenAI-compatible Chat Completions;
- OpenAI-compatible Responses-style item and tool flows;
- model listing and capability discovery;
- embeddings and reranking where a declared model supports them;
- tool calls and structured outputs;
- streaming events;
- state/session endpoints;
- artifact/profile inspection; and
- a native API for Hawking-specific context, Passport, and Doctor data.

Compatibility is a product surface, not a reason to flatten Hawking’s richer state model internally.

### 10.4 MCP in both directions

Hawking should act as:

- an MCP client that consumes local or permitted remote tools/resources/prompts; and
- an MCP server that exposes Hawking artifacts, context resources, workspace indexes, session state, and Doctor/Passport inspection to other clients.

Tools are discovered and summoned under permission and token budgets.

### 10.5 Client compatibility

A user should be able to point existing OpenAI-compatible or MCP-capable clients at Hawking without custom model conversion.

HIDE remains the highest-fidelity reference client, but compatibility should include coding agents, notebooks, scripts, automation systems, and local chat applications.

### 10.6 Artifact adapter ABI

A runtime adapter declares:

- supported artifact/container version;
- supported representation and treatment operators;
- model architectures;
- tokenizer/chat-template support;
- context policies;
- state codecs;
- device/backend;
- numerical parity contract;
- resident expansion;
- feature limitations; and
- source hash.

Unsupported operators fail closed. No adapter may silently reconstruct a dense parent and still claim the Hawking artifact’s physical deployment benefits.

### 10.7 No lock-in

Hawking must make it easy to:

- export the Passport;
- export or rederive compatible artifacts;
- inspect every model-specific byte;
- move a session capsule;
- replace HIDE;
- replace the runtime where another backend is stronger;
- use standard tools and protocols; and
- reproduce the artifact without a hosted Hawking service.

The moat is verified transformation knowledge and Doctor intelligence, not hostage-taking.

## 12. Current Hawking grounding

This frontier should build from existing Hawking concepts rather than create a disconnected second project.

### 12.1 Existing context vocabulary

The repository already defines context profiles with:

- native, PI, YaRN, and Dynamic NTK position policies;
- lossless and bounded working sets;
- lossless, StreamingLLM-like, SnapKV-like, and H2O-like eviction choices;
- native, F16, and INT4 KV precision;
- source weighting;
- recency half-life;
- head/tail ordering;
- retrieval count; and
- iterative context improvement.

The required work is to turn those configuration surfaces into measured end-to-end behavior and receipts.

### 12.2 Existing context-extension probe

The existing long-context probe already states the correct three-part logic:

- post-hoc position extension;
- KV memory as the practical wall; and
- recurrent/SSM state as a flat-memory alternative.

It also states the critical product connection: smaller weights leave more KV headroom. This frontier turns that probe into a validated product path.

### 12.3 Existing HIDE and agent substrate

The repository contains real or partially built concepts for:

- event logs;
- permissions;
- persistence;
- tool dispatch;
- agent FSM state;
- MCP;
- context profiles;
- workspace/index abstractions;
- OpenAI-compatible serving; and
- structured decoding.

The repository’s own audits also identify that much of the agent loop, context compiler, code index, host integration, and API wiring is shallow or disconnected.

This is favorable. The frontier is primarily a **connection, hardening, measurement, and UX program**, not a reason to restart from zero.

### 12.4 Non-interference

This ecosystem generation must remain additive while the live Doctor campaign is active.

It may:

- create new documents;
- build isolated context and agent tests;
- define new schemas;
- wire read-only artifact inspection;
- implement APIs against sealed fixtures;
- build HIDE features in isolated branches; and
- prepare successor runtime profiles.

It may not:

- change active cell recipes;
- change rate/branch semantics;
- rewrite completed observations;
- mutate run-bound files;
- delete source or results outside lifecycle authority;
- consume resources that destabilize the campaign; or
- activate new defaults before the supersession gate.

## 16. Hard prohibitions

Hawking must not:

- use retrieval or tools to claim a failed model was restored;
- market nominal context as usable context;
- call summarized state exact recall;
- omit KV, index, session, slice, or resident-expansion bytes from device comparisons;
- compare a Hawking system with rich tooling against an ordinary quant with tools disabled;
- let a remote model silently answer a local-agent task;
- hide network egress;
- preload an enormous tool catalog and call it context efficiency;
- store unproven memories as permanent facts;
- let summaries erase user constraints, failures, or permissions;
- call an event log resumable without reboot/crash proof;
- let HIDE become required for artifact use;
- reconstruct a dense parent inside a compatibility adapter while preserving the compact-deployment claim;
- ship a progressive slice without exact dependency and byte receipts;
- change active Doctor runs or current progress to accelerate this ecosystem work;
- use serving speed to excuse model or context failure;
- claim “unbounded context” without specifying exact, paged, retrieved, or summarized semantics;
- optimize for one flashy NIAH result while failing code, multi-hop, or agent continuity;
- expose destructive tools without a deterministic permission boundary; or
- build a closed proprietary protocol where an open compatibility surface can preserve Hawking’s semantics.
