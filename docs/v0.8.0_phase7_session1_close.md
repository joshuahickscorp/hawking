## What's still in F32 (sessions 2-4)

The "bridge" kernels mean only the rmsnorm-gemv fusion sites read f16.
The rest of the residual stream is still f32. To capture the full
1.3-1.5× Phase 7 gain, future sessions need:

- Session 2: f16 variants of mla_decode_kernel + o_proj
- Session 3: f16 LM head (gemv_f16 already exists; needs f16-input
  variant wired to the default path)
- Session 4: residual stream f16 throughout (every layer's rmsnorm
  output → next layer's input is f16, no f32 conversion)

Per-routed-expert F16 bridge is also deferred: the variable expert
selection makes per-call batching complex. Session 2 scoping should
revisit the MoE routed path.

## Expected gain from Session 1 alone

Modest. The bridge kernels read f16 instead of f32 for ONE input per
fused dispatch. Bandwidth saved: ~2KB per kernel call vs ~4KB. With
~54 fused dispatches per token (27 attn + 27 ffn shared), that's
27 × 2KB + 27 × 4KB = ~162KB/token saved. At memory bandwidth ~100 GB/s,
that's ~1.6 µs/token saved — well under 1% of decode wall.

The structural change matters more than the perf number. Session 1
is the foundation that lets sessions 2-4 each capture 8-15% gain by
eliminating their respective f32 conversions.
