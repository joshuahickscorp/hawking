# v0.2.3 — per-dispatch profile data

## Methodology

`DISMANTLE_TRACE_DISPATCH=1` enables per-dispatch wall-clock timing inside
`MetalContext::dispatch_threads` and `MetalContext::dispatch_batch`. Each call
records a `DispatchSample { kernel_name: &'static str, wall_us: u64, layer_hint: Option<u32> }`.
Overhead: see sub-task 7 result in `reports/v0.2.3_instrumentation.md` — negligible on the
default path (env var absent).

Three configs profiled at 64-token decode, 1 trial, `--kernel-profile
profiles/deepseek-v2-lite-q4.m3pro18.json` as the base.

**Note on Config 2**: The `sed` command in sub-task 4 was designed to produce a v0.1.2-style
baseline (no Metal MLA, no decode-arena) by changing `selected.id` to `gpu-greedy-frontier`.
In practice it only patched the metadata `id` field; the actual schedule fields
(`moe_schedule`, `mla_schedule`, `gpu_buffer_reuse`) were unchanged. Config 2 therefore ran the
same kernel path as Config 1 and is excluded from independent analysis. The numbers confirm
this: 1.724 vs 1.721 dec_tps, ~identical dispatch distribution.

**Note on `other` bucket**: `static_kernel_name()` in `metal/mod.rs` maps 18 known names.
Calls that don't match fall through to `"other"`. The `other` bucket is the largest single
category (42% of dispatch wall); resolving it requires extending the static map to cover all
`_indexed`, attn, and rope kernel variants. This is a follow-on task for v0.3.

## v0.2.2-metal-safe profile (1.721 dec_tps)

- total dispatches/token: 466
- total dispatch wall/token: 305 ms (52.4% of decode wall)

| kernel | count | total_us | mean_us | % of dispatch |
|---|---|---|---|---|
| other | 16,048 | 8,229,692 | 512 | 42.2% |
| dispatch_batch | 880 | 6,389,268 | 7,260 | 32.8% |
| gemv_f32_attn | 5,508 | 2,152,908 | 390 | 11.0% |
| rmsnorm | 5,576 | 1,067,810 | 191 | 5.5% |
| mla_decode_kernel | 1,836 | 1,001,216 | 545 | 5.1% |
| gemv_f16 | 4 | 660,457 | 165,114 | 3.4% |

Dispatch breakdown: 880 `dispatch_batch` calls / 64 tokens = 13.75 batches/token, consistent
with 12 MoE layers out of 27 total (each token hits ~14 of the 12 MoE layers plus partial
overlap with layer indexing).

Per-layer observations:
- **Layer 1 anomaly**: 2.62 s total (40,929 µs/token), dominated by `dispatch_batch` (91%).
  This is first-dispatch JIT pipeline compile latency on the Metal device. Layers 2–26 are
  normal. This single warmup layer accounts for 13.4% of total dispatch wall.
- **MoE layers (12/27)**: avg ~7.1 ms/token (excluding L1 anomaly). `dispatch_batch` is the
  single timing unit — each call encodes gate + up + silu + down + accumulate for all 6
  selected experts in one command buffer.
- **Dense layers (15/27)**: avg ~10.8 ms/token. `other` bucket dominates (80–81% per dense
  layer). These are likely attn/rope/kv-cache kernels not yet in `static_kernel_name`.
- Layer 16, 10, 19, 22 are the most expensive dense layers (~12 ms/token each).

## v0.2.0-metal-all profile (0.740 dec_tps — the regression case)

Config: `moe_schedule=two-stage`, `mla_schedule=metal-mla`, `gpu_buffer_reuse=decode-arena`.
Produced 35 tokens before EOS (numerical divergence from two-stage path, as seen in v0.2.1
diagnostic).

- total dispatches/token: 489 (17,121 samples / 35 tokens)
- total dispatch wall/token: 573 ms (42.4% of decode wall)

| kernel | count | total_us | mean_us | % of dispatch |
|---|---|---|---|---|
| dispatch_batch | 503 | 9,506,129 | 18,898 | 47.4% |
| other | 9,204 | 5,253,655 | 570 | 26.2% |
| gemv_f32_attn | 3,159 | 3,441,019 | 1,089 | 17.2% |
| mla_decode_kernel | 1,053 | 994,065 | 944 | 5.0% |
| rmsnorm | 3,198 | 688,551 | 215 | 3.4% |
| gemv_f16 | 4 | 157,330 | 39,332 | 0.8% |

Key delta vs v0.2.2-metal-safe:

- `dispatch_batch` mean: **18,898 µs vs 7,260 µs → 2.6× slower per batch**
- `dispatch_batch` fraction of total dispatch wall: 47.4% vs 32.8%
- `gemv_f32_attn` mean: 1,089 µs vs 390 µs → 2.8× slower per call

The `dispatch_batch` slowdown is the root cause. Two-stage MoE materializes intermediate
results to device memory between stages, requiring the GPU to complete a full round-trip before
the output stage can read its inputs. The `indexed-no-pack-one-cb` path encodes the entire
MoE block (gate → router → up/gate gemm → silu → down gemm → accumulate) in a single command
buffer; the GPU can pipeline the stages internally without round-trips.

`gemv_f32_attn` slowdown under two-stage is secondary and possibly a timing artifact: with
fewer tokens produced and less GPU steady-state utilization, the GPU may be at lower clock
when attention runs.

## Findings

1. **`other` bucket blocks further analysis.** At 42% of dispatch wall in the safe config, it
   is the single largest time category. Until `static_kernel_name` is extended to cover the
   indexed MoE kernel variants and the attn/rope/kv kernel names, we cannot determine whether
   the dominant cost is attention, FFN, or something else.

2. **`dispatch_batch` is the correct unit of analysis for MoE.** The entire MoE block
   (6 experts × 4 sub-kernels + gate + accumulate) commits in one Metal command buffer and
   is recorded as one sample. Mean = 7,260 µs under indexed-one-cb. This is the floor for any
   MoE block optimization — a rewrite must beat 7.26 ms to help.

3. **Two-stage MoE regression is fully explained.** The 2.6× `dispatch_batch` slowdown is the
   difference between a single-pass batched kernel (indexed-one-cb) and a two-pass
   materialized intermediate (two-stage). No other kernel shows a comparable shift.

4. **Dense layers are currently more expensive than MoE layers per token** (~10.8 ms vs
   ~7.1 ms excluding L1 warmup). This is counterintuitive — but `other` dominates dense
   layers, so these are likely attention + rope + kv operations that haven't been resolved.

5. **JIT warmup is a bench artifact.** Layer 1 adds 2.62 s to total dispatch wall (13% of
   sample), which is a one-time Metal pipeline compile cost. Multi-trial benches absorb this;
   single-trial runs (as in sub-task 4) don't. The warmup is not a real inference cost.

6. **52% of decode wall is in dispatches (safe config).** The other 48% is CPU overhead:
   tensor reshape, KV-cache writes, sampling, and Rust runtime costs. Neither pure kernel
   optimization nor pure dispatch-count reduction can close the full gap without addressing
   both halves.

---

## Implications for v0.3 wedge design

**Prerequisite before any v0.3 wedge**: extend `static_kernel_name` to cover all real kernel
names and re-profile. The `other` (42%) and the true dense-layer breakdown are unknowns.

Given what is resolved:

- **Highest-ROI target: fuse the dense-layer attention stack.** Dense layers take ~10.8 ms/tok
  vs MoE layers at ~7.1 ms/tok. If `other` in dense layers is primarily attention (qkv proj +
  rope + softmax + kv-cache write + output proj), a single fused FlashAttention-style kernel
  would collapse 5–6 dispatches into 1 and eliminate intermediate buffers. This is the v0.3
  candidate with the highest ceiling.

- **Do not revisit two-stage MoE.** The 2.6× `dispatch_batch` slowdown is structural: it
  requires device-memory round-trips that the hardware cannot pipeline. `indexed-one-cb` is
  already optimal for the MoE block shape at batch=1.

- **Dispatch count reduction has limited upside.** At 466 dispatches/token, even eliminating
  10% of calls saves <2 ms/token. The bottleneck is kernel execution time, not launch overhead.

- **layer-CB (layer-level command-buffer batching) is still unvalidated.** The v0.2.1
  diagnostic saw −18% from layer-CB. Once `other` is resolved and dense-layer costs are known,
  revisit whether batching attn + FFN + norm in one CB helps or hurts on Apple Silicon.
