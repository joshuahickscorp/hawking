# v0.5.0 Dead Ends — Lessons from Failed Wedges

Catalog of approaches the v0.3.x campaign tried and abandoned. Documenting
these so future wedges don't repeat the same mistakes.

## v0.3.7 — full-expert CB fusion (parity failed)

**Stashed at**: `git stash list` → "v0.3.7 STUCK"
**Goal**: fuse gate + up + silu + down per-expert into one CommandBatch.
**Failure mode**: parity tests failed (~1e-2 diff vs reference at realistic
shapes). The composed GEMV+silu+GEMV chain amplifies Q4_K dequant error
beyond what the existing ATOL_FULL=1e-2 budget allows for hidden=2048.

**Compounding issue (Issue 3 from v0.3.7 STUCK report)**: the per-expert
fallback path the wedge targeted is DEAD CODE for production. The default
moe_schedule="indexed-no-pack-one-cb" routes through
`moe_block_batched_dispatch` which returns early before the per-expert
fallback. Even if parity were fixed, structural metrics wouldn't change.

**Lesson**: when adding a new fusion path, verify the production hot path
actually fires it (per v0.3.9 audit). The audit caught this; the v0.3.7
plan ignored it.

**Future revisit**: tractable if wedge targets `moe_block_batched_indexed_metal`
(the actual hot path) AND uses scaled-down synthetic weights so silu output
magnitude stays bounded.

## v0.3.8 — gemv_f16 multirow TG (regressed -12%)

**Stashed at**: `git stash list` → "v0.3.8 STUCK"
**Goal**: 8 output rows per threadgroup with shared-memory x-cache for
the LM head GEMV (vocab × hidden = ~100k × 2048).
**Failure mode**: kernel ran 12% SLOWER than scalar (377k µs vs 337k mean).
Parity passed (atol < 1e-3) — kernel is correctness-clean.

**Root cause hypothesis** (not investigated):
- Cooperative shmem load of x (256 threads writing 2048 floats with one
  barrier) costs more than 102400 TGs each independently reading their cols.
- M3 Pro threadgroup memory bank conflicts when 32 lanes read same shmem index.
- 256-thread TGs with mostly-idle simdgroups (8 rows/8 simdgroups) underutilize.

**Lesson**: multi-row TG with shmem caching is not automatically faster on
Apple Silicon. Per-row independent reads can outperform when DRAM cache
serves them well (LM head hits the same x for every row).

**Future revisit**: profile with Xcode GPU Frame Capture to see which
barrier or memory pattern is the bottleneck. simdgroup_matrix on the LM
head specifically (since vocab × hidden is matmul-shaped) is also worth
trying.

## v0.5.0-fused-arena (single-kernel MoE, 190× slower)

**Reverted in commit**: 683888b (kept in profile.rs at deterministic_rank=50)
**Goal**: activate moe_block_fused_v2lite_indexed_metal as default MoE path.
**Failure mode**: production model (DeepSeek-V2-Lite) generates at 0.01
dec_tps (190× slower than 1.91 baseline). Parity tests pass — kernel
correct in isolation.

**Root cause**: the fused MoE kernel was designed for uniform Q4_K_M
weights. DeepSeek-V2-Lite has mixed-quant down_proj — Q8_0 routed, Q6_K
shared (per v0.3.9 hot-path discovery). The dispatch's dtype guards reject
this model and the call falls through to the per-expert CPU fallback.

**Lesson**: kernel-level parity is necessary but not sufficient. Always
verify the dispatch path actually fires for the production model AND the
weights match the kernel's dtype contract.

**Future revisit**: extend `moe_block_fused_v2lite_indexed_metal` to
support mixed-quant down (separate kernel paths for Q8_0 vs Q6_K down).
Or: revisit when fp16 activations land (Phase 7) — the residual precision
change might allow the fused kernel to handle dequant differently.

## simdgroup_matrix wrong primitive for GEMV (v0.3.0)

**Reverted**: v0.3.0 attempt regressed -10.6% from v0.2.2 baseline.
**Goal**: rewrite `gemm_q4_k_m_fused` using simdgroup_matrix<8,8>.
**Failure mode**: 64 threadgroup barriers per row inner loop, broadcast
wasted 7/8 columns. simdgroup_matrix is for matmul, not GEMV.

**Lesson**: simdgroup_matrix's strength is in real GEMM (M×N×K matmul where
all three dims are non-trivial). For GEMV (K-dim only), `simd_sum` is the
right primitive — zero barriers, single shuffle reduction. v0.4.0
implemented this correctly and got 1.23× per kernel.

**Future revisit**: the LM head GEMV is the only place in the model that's
shaped like a real GEMM (vocab × hidden = 102k × 2048). Worth considering
simdgroup_matrix there specifically — but per v0.3.8, the obvious multi-row
TG approach didn't beat scalar, so this needs careful design.
