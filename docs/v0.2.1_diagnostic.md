Bench run: 2026-05-03. 5 configs × 3 trials × 64 max-new-tokens, "Once upon a time"
prompt, temp=0 greedy, M3 Pro 18 GB, slm not running.

**Note on token counts.** two_stage (57) and all (35) diverge from baseline (64) before
`max_new_tokens` is reached. The model hit an EOS token earlier on each numerically-different
computation path. The all vs two_stage tps comparison is confounded: "all" runs a shorter
sequence (35 tokens, smaller KV cache per attention call) so its per-token overhead is lower
independent of wedge benefits. Direct per-wedge attribution between those two is unreliable.

## Per-dispatch analysis

`--trace-json` emits **high-level timing only**: decode_tps, target_gap_x, rss_mb_after_run,
and the full bench report. No per-kernel timestamps, no command-buffer timing, no
per-dispatch breakdowns.

**Tractability of adding per-dispatch timing:** moderate effort, ~50–100 lines of Rust.
The approach: wrap each `commit()` + `wait_until_completed()` call in `kernels/mod.rs` with
`std::time::Instant` before/after, accumulate per-kernel wall times into a
`Vec<(label, duration_ms)>` in `GenerateStats`, and emit them in `trial_stats` per bench trial.
Since `wait_until_completed()` is already a blocking sync point, the CPU-side timestamps
directly capture GPU execution + launch overhead with no extra sync cost. This would let a
future diagnostic isolate exactly how much of the 2782ms/token in two_stage is dispatch
overhead vs kernel arithmetic time.

The current bisect data answers the wedge attribution question clearly enough that per-dispatch
instrumentation is not required for the v0.2.2 fix decision.

## Root-cause hypothesis confirmation

Closeout hypothesis: *per-layer GPU commit+wait overhead from Metal MLA's added dispatches.*

### Prediction 1 — mla alone shows a regression

**FAIL.** Metal MLA without layer-cb produces 1.64 dec_tps vs 1.70 baseline = −4%, within
normal measurement variance (the first trial of baseline was 1.12, indicating high run-to-run
noise at model load time). Metal MLA does not cause meaningful decode regression at 64-token
sequences when each dispatch is independent (moe-block command_buffering).

### Prediction 2 — mla_cb regression < mla regression (Layer-CB helps but doesn't fully recover)

**FAIL.** Adding layer-cb to Metal MLA makes things *worse*: 1.34 vs 1.64 dec_tps (−21% vs
−4%). The `mla_decode_and_o_proj_metal` kernel batched via `dispatch_batch` appears to carry
higher overhead than two separate `mla_decode_metal` + `gemv_f32_attn_dispatch` calls at
batch=1. One possible cause: the batched command buffer may trigger additional JIT
recompilation on first use; the first trial for mla_cb is 0.70 tps (vs 1.02 for mla alone).
By the third trial mla_cb reaches 1.58 — comparable to mla alone at 1.72 — suggesting warmup
cost, not structural overhead. With 3 trials the median is dragged by the slow first trial.

### Prediction 3 — two_stage regression is independent of the MLA-related regressions

**PARTIAL.** two_stage alone is indeed the dominant regression source and its cause is
independent of the MLA changes (CPU MLA is unchanged in config 3). However, the wedges are
not strictly additive: "all" at 0.49 tps is faster than "two_stage" at 0.36 tps despite
containing more wedges. This is confounded by the token-count divergence (35 vs 57 tokens;
shorter sequence = faster CPU MLA per token via smaller KV cache), so this sub-additivity
does not necessarily indicate synergy between wedges.

### Closeout hypothesis verdict: REJECTED

The closeout attributed the regression to Metal MLA's 27 GPU roundtrips per token. The data
shows Metal MLA alone contributes ≤4% regression — near noise. The regression is concentrated
in two-stage MoE (Wedge 2): 2782 ms/token vs 588 ms/token baseline = **4.7× slower per
token from that wedge alone**.

## Two-stage MoE root cause (inferred from perf data, no code change made)

At decode batch=1, the two-stage MoE design materializes all expert intermediates to device
memory between Stage 1 and Stage 2. DeepSeek-V2-Lite runs 27 MoE layers per token. At
batch=1 with top-k expert selection, the arithmetic per layer is very small; the intermediate
write-to-device / read-from-device round-trip dominates. The `indexed-no-pack-one-cb` path
avoids this by computing the full expert output in one kernel with no intermediate
materialization. This penalty does not appear in the closeout's bench (which only ran the
combined all config) because that bench confirmed correctness + composite tps, not per-wedge
attribution.

## Recommendation for v0.2.2

**Revert two-stage MoE to `indexed-no-pack-one-cb` as the default decode schedule.**

The regression is concentrated in Wedge 2 (two-stage MoE). Reverting that wedge alone should
recover to ~1.64–1.70 dec_tps (mla_cb warmup noise aside). Wedges 1, 3, 4 (Metal MLA,
layer-cb, decode-arena) do not independently cause the regression and can remain as-is or as
opt-in research variants.

Secondary: the mla_cb warmup penalty (trial 1 at 0.70 tps vs 1.58 for warm trials) deserves
investigation. If the `mla_decode_and_o_proj_metal` kernel is triggering JIT recompilation on
first use, pre-warming the command buffer at model load time would close that gap.

Do **not** attempt to "fix" two-stage MoE in v0.2.2 by adding an intermediate-buffer pool
without first profiling the device-memory write/read cost with per-dispatch timing. The fix
may be: defer two-stage MoE to prefill-only contexts (batch > 1 or seq_len > 512) where the
intermediate materialization is amortized.
